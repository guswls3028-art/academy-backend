## 1. 배포 전 체크

- **Docker 순서**: academy-base → api → video-worker → ai-worker → messaging-worker. base 없으면 실패.

- **API 기동 전**: migrate, 필수 env(DB_*, R2_*, SQS, INTERNAL_WORKER_TOKEN), GET /health 200
- **워커**: `DJANGO_SETTINGS_MODULE=apps.api.config.settings.worker`, API_BASE_URL·INTERNAL_WORKER_TOKEN

## 2. 엑셀 등록 흐름

- API: 파일 → R2 업로드 → SQS 등록 → 202 + job_id
- 프론트: 학생 등록은 `/api/v1/students/excel_job_status/<job_id>/`, 수강 등록은 `/api/v1/enrollments/excel_job_status/<job_id>/` 폴링 → DONE 시 결과
- 워커: SQS → R2 다운로드 → `ExcelParsingService` → `lecture_id` 있으면 `lecture_enroll_from_excel_rows`, 없으면 `import_students_from_rows` → R2 삭제
- 학생 import row 정책 SSOT: `apps.domains.students.services.import_students`. 학생-only Excel과 수강등록 Excel 모두 여기서 중복/복원/생성 판단을 공유한다.

## 7. 비디오 인코딩이 진행되지 않을 때

현재 영상 인코딩 운영 절차는 [runbooks/video-batch.md](runbooks/video-batch.md)를 우선한다. Video daemon/SQS/ASG 경로는 2026-05-10 폐기됐고, 현재 운영 경로는 API `upload_complete` → `VideoTranscodeJob` → AWS Batch `SubmitJob`이다.

### 7.1 Batch 큐와 job definition 확인

```powershell
cd C:\academy\backend
pwsh scripts/v1/verify-video-batch-connection.ps1 -Profile default
```

직접 확인할 때는 현재 SSOT 이름을 사용한다.

```powershell
aws batch describe-job-queues --job-queues academy-v1-video-batch-queue --region ap-northeast-2 --query "jobQueues[0].{status:status,state:state}" --output table
aws batch describe-job-definitions --job-definition-name academy-v1-video-batch-jobdef --status ACTIVE --region ap-northeast-2 --query "jobDefinitions[-1].{revision:revision,status:status}" --output table
aws batch list-jobs --job-queue academy-v1-video-batch-queue --region ap-northeast-2 --job-status RUNNING --output table
```

### 7.2 API upload_complete 503 확인 순서

- R2 source object 확인과 presigned GET 생성 여부를 본다.
- API 컨테이너 env에 `VIDEO_BATCH_JOB_QUEUE=academy-v1-video-batch-queue`, `VIDEO_BATCH_JOB_DEFINITION=academy-v1-video-batch-jobdef`가 반영됐는지 본다.
- API IAM 역할에 Batch `SubmitJob`, `DescribeJobs`, `TerminateJob` 권한이 있는지 본다.
- `VideoTranscodeJob` 상태가 stuck이면 `scan_stuck_video_jobs`, `reconcile_batch_video_jobs` 운영 명령을 사용한다.

### 7.3 폐기된 Video SQS/ASG 신호

- `academy-video-jobs`, `VIDEO_SQS_QUEUE_NAME`, `academy-video-worker-asg`, `BacklogCount`, `VIDEO_BACKLOG_API_URL`은 daemon/SQS 시절 잔재다.
- `academy-worker-queue-depth-metric` Lambda와 `academy-worker-queue-depth-rate` EventBridge rule은 2026-05-25 KST live check 기준 운영 계정에 없다. AI/Messaging scale은 AWS/SQS CloudWatch alarm → EC2 ASG StepScaling 경로를 본다.

## 8. 업로드 완료 503 / "업로드 완료 처리 중 오류가 발생했습니다"

> **2026-05-25 정정:** 아래 장애의 현재 경로는 Video SQS가 아니라 AWS Batch다.
> `upload_complete`는 R2 source 확인, presigned GET, ffprobe, `VideoTranscodeJob`
> 생성 후 `submit_batch_job`을 호출한다. Video daemon/SQS 문맥의
> `deploy_worker_asg.ps1`, `sync_api_env_lambda_internal.ps1`,
> `VIDEO_SQS_QUEUE_NAME` 안내는 현재 실행 진실이 아니다.

**현재 원인 요약**: `upload_complete`가 503을 반환하면 우선 R2 source 확인,
presigned GET, ffprobe, Batch submit, API SSM env 반영 상태를 본다.

### 8.1 왜 발생하는가

- Messaging/AI worker env 갱신은 `scripts/v1/update-workers-env-sqs.ps1`, API env 재반영은 `scripts/v1/refresh-api-env.ps1`가 현재 스크립트다.
- **API 서버(academy-api 컨테이너)** 는 SSM을 주기적으로 다시 읽지 않음.  
  → API는 컨테이너 생성 시점의 `/opt/api.env`만 사용하므로 SSM 변경 후 컨테이너 재생성이 필요하다.
- API는 `create_job_and_submit_batch` → `submit_batch_job`을 호출하므로 `VIDEO_BATCH_JOB_QUEUE`, `VIDEO_BATCH_JOB_DEFINITION`, AWS region/권한이 필요하다.
- R2 확인 또는 presigned URL 생성 실패도 `_upload_complete_impl`에서 503/409로 드러날 수 있다.

필요한 env 예시:

- `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` (또는 EC2 인스턴스 프로필)
- `AWS_DEFAULT_REGION` 또는 `AWS_REGION` (예: `ap-northeast-2`)
- `VIDEO_BATCH_JOB_QUEUE`, `VIDEO_BATCH_JOB_DEFINITION`

### 8.2 해결: API 쪽 .env 동기화

SSM을 올린 뒤 **API 서버에 반영**하려면 (로컬 PowerShell에서):

```powershell
pwsh scripts/v1/refresh-api-env.ps1 -AwsProfile default
```

- SSM `/academy/api/env` → API EC2 `/opt/api.env` 반영
- API 컨테이너 재생성

Worker SSM의 Messaging/AI 큐 이름만 갱신해야 한다면:

```powershell
pwsh scripts/v1/update-workers-env-sqs.ps1 -AwsProfile default
```

#### 8.2.1 .env 수정 후 컨테이너 재생성 (필수)

`docker restart` 는 컨테이너를 그대로 둔 채 재시작만 하므로, **생성 시점에 읽은 .env만** 사용한다. .env를 수정·추가한 뒤에는 **컨테이너를 지우고 다시 만들어야** 새 env가 반영된다.

**API 서버(ec2-user)에서 한 번에 실행 (복붙):**

```bash
IMG=$(docker inspect academy-api --format '{{.Config.Image}}' 2>/dev/null); [ -z "$IMG" ] && IMG=$(docker images --format '{{.Repository}}:{{.Tag}}' | grep academy-api | head -1); docker stop academy-api 2>/dev/null; docker rm academy-api 2>/dev/null; docker run -d --name academy-api --restart unless-stopped --env-file /home/ec2-user/.env -p 8000:8000 "$IMG"
```

정식 경로는 `scripts/v1/refresh-api-env.ps1`이며, 수동 docker 명령은 긴급 복구용으로만 사용한다.

#### 8.2.2 복구: .env 날아갔을 때 (빠른 복구)

**API가 쓰는 env 이름 (코드 기준)**  
DB_NAME, DB_USER, DB_PASSWORD, DB_HOST, DB_PORT  
R2_ACCESS_KEY, R2_SECRET_KEY, R2_ENDPOINT, R2_PUBLIC_BASE_URL  
REDIS_HOST, REDIS_PORT, REDIS_PASSWORD(선택), REDIS_DB  
AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION (또는 AWS_DEFAULT_REGION)  
VIDEO_BATCH_JOB_QUEUE, VIDEO_BATCH_JOB_DEFINITION, INTERNAL_WORKER_TOKEN, SECRET_KEY, LAMBDA_INTERNAL_API_KEY(선택)

**방법 1 – 로컬에 풀 .env 있음**  
PowerShell: `pwsh scripts/v1/refresh-api-env.ps1 -AwsProfile default` 실행.

**방법 2 – 서버 .env만 복구 (EC2에서)**  
SSM으로 덮어쓸 때 반드시 **탭을 줄바꿈으로** 바꿔야 함. `> .env` 만 하면 한 줄로 저장돼서 Docker가 env를 못 읽음.

```bash
cd /home/ec2-user/academy
# SSM → .env (탭→줄바꿈 적용, 이거 안 하면 .env 한 줄로 깨짐)
bash scripts/fetch_ssm_to_env.sh
# 컨테이너 재생성 (restart 아님)
bash scripts/refresh_api_container_env.sh
```

SSM에 R2/AWS 키가 없으면 .env에 추가한 뒤 다시 `refresh_api_container_env.sh` 실행.  
**앱이 쓰는 변수명**: `R2_ACCESS_KEY`, `R2_SECRET_KEY`, `R2_ENDPOINT`, `R2_PUBLIC_BASE_URL` (CLOUDFLARE_R2_*, VIDEO_BUCKET_NAME 아님).

### 8.3 1분 컷: API 컨테이너 안에서 Batch 설정 확인

API 서버(EC2)에 SSH 후:

```bash
sudo docker exec -it academy-api bash
```

컨테이너 안에서:

```bash
python - << 'PY'
import os
import boto3
region = os.getenv("AWS_DEFAULT_REGION") or os.getenv("AWS_REGION", "ap-northeast-2")
queue_name = os.getenv("VIDEO_BATCH_JOB_QUEUE", "academy-v1-video-batch-queue")
job_def = os.getenv("VIDEO_BATCH_JOB_DEFINITION", "academy-v1-video-batch-jobdef")
print("REGION:", region)
print("BATCH_QUEUE:", queue_name)
print("BATCH_JOB_DEF:", job_def)
batch = boto3.client("batch", region_name=region)
print(batch.describe_job_queues(jobQueues=[queue_name])["jobQueues"][0]["status"])
print(batch.describe_job_definitions(jobDefinitionName=job_def, status="ACTIVE")["jobDefinitions"][-1]["revision"])
PY
```

**해석**

- 여기서 예외 나면 → upload_complete 503 원인으로 **Batch/자격증명/리전 문제** 가능성이 높다.
- `InvalidClientTokenId` / `SignatureDoesNotMatch` → 자격 증명(키/시크릿) 확인.
- `ClientException` / job queue 없음 → Batch queue/job definition 이름·리전 확인.
- `Could not connect to the endpoint URL` → 네트워크/리전/엔드포인트 확인.
- `KeyError` → env에서 `VIDEO_BATCH_JOB_QUEUE`, `VIDEO_BATCH_JOB_DEFINITION`, 리전이 비어 있을 수 있음.

### 8.4 Lambda timeout과의 관계

- legacy Video backlog Lambda/ASG는 현재 운영 판단의 SSOT가 아니다.
- 현재 영상 처리 상태는 Batch queue/job, `VideoTranscodeJob`, `scan_stuck_video_jobs`, `reconcile_batch_video_jobs`로 확인한다.
