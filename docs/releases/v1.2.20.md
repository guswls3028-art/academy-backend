# V1.2.20 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-02 KST
**Scope:** OMR v2 sheet contract pipeline refactor

This release turns the OMR sheet shape into an explicit runtime contract.
Rendering, worker payloads, answer persistence, recognition facts, and grading
readiness now share the same `OMRSheetContract` instead of each path
interpreting `total_questions` independently.

## 1. Root Cause

- Previous fixes separated objective and essay counts, but the mapper still
  built `question_number -> ExamQuestion.id` from every question on a sheet.
- A worker response containing q21~q25 for a `20 objective + 5 essay` sheet
  could still be mapped to essay `ExamQuestion` rows as if they were objective
  answers.
- Recognition facts did not store the exact sheet contract used at recognition
  time, so post-incident audits had to infer the expected objective count again.

## 2. Fix

- Added pure domain contract objects:
  - `academy/domain/omr/contract.py`
  - `apps/support/omr/contract_builder.py`
- Replaced OMR runtime shape resolution with `OMRSheetContract` in:
  - dispatch payload building
  - OMR document defaults
  - deprecated `generate-omr` compatibility view
  - mapper exam structure loading
  - recognition fact persistence
  - grading readiness
- Added `OMRRecognitionRun.contract_snapshot` via migration
  `submissions.0006_omrrecognitionrun_contract_snapshot`.
- `SubmissionAnswer` persistence now stores only contract-valid objective
  question numbers.
- Worker answer count checks now compare both raw answer count and unique
  mapped objective question count to the contract.
- Worker q21+ values for mixed sheets are preserved in `OMRDetectedAnswer`
  facts but are not written to compatibility `SubmissionAnswer` rows.

## 6. Notes

- This release still does not auto-grade essay text. It makes the objective
  recognition contract explicit and auditable so mixed sheets cannot be treated
  as all-objective sheets again.
