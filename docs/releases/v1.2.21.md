# V1.2.21 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-02 KST
**Scope:** multi-domain hidden failure hardening + production verification closure

This release closes a broad audit slice where code could pass local checks while
the real service still behaved incorrectly. It fixes production failure paths
across subscription privacy, submission tenant isolation, result scoring, async
AI callbacks, video enqueue failure handling, messaging idempotency, and
frontend async/status UX contracts.

## 1. Root Cause Themes

- Some paths were technically valid in tests but not safe in real user flows:
  anonymous subscription responses exposed staff billing fields, Enter key score
  entry could submit duplicate PATCHes, and "queued" message wording implied
  successful delivery before provider completion.
- Several domain boundaries had duplicated or incomplete contract handling:
  AI job terminal statuses were defined in more than one frontend module, and
  submission status `superseded` was not represented consistently.
- Worker completion order was unsafe: domain callback failure could previously
  be swallowed or could keep AI jobs in `RUNNING` without preserving terminal
  results, causing retry semantics to be misleading.
- Some score and result paths treated sentinel states as normal failures:
  `NOT_SUBMITTED` attempts could inflate fail counts, and manual item-score
  entry did not clear the not-submitted state.

## 2. Fix Summary

- Anonymous `GET /api/v1/core/subscription/` now hides billing-only fields unless
  the requester is effective tenant staff.
- Submission creation now validates both target exam tenant and enrollment
  membership through tenant-scoped exam/session joins.
- Manual item-score entry clears `NOT_SUBMITTED` so a real score is reflected
  as an attempted result.
- Session score summaries now group attempts by `(exam_id, enrollment_id)` and
  exclude `NOT_SUBMITTED` from fail counts.
- AI SQS worker now persists terminal AI job state/results before domain
  callback, keeps the SQS message when callback fails, and on terminal redelivery
  retries the callback without rerunning inference.
- Domain AI callback dispatch now reports handler failure to the worker instead
  of always swallowing it as success.
- Video upload completion now marks true Batch submit failure as `FAILED` and
  returns 503 instead of presenting an unqueued upload as accepted.
- Messaging manual occurrence fallback now uses a UUID-based key rather than
  second-level timestamps.
- Frontend status contracts now share one AI job terminal-status SSOT, recognize
  `superseded` submissions, prevent duplicate mobile score submit, validate rich
  text by visible content, and use less misleading message-send wording.
