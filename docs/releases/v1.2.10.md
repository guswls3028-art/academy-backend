# V1.2.10 Release Notes

**Status:** production-deployed  
**Release date:** 2026-05-23 KST  
**Scope:** Clinic participant write SSOT + student clinic change UX contract

## 1. Clinic Participant Write SSOT

- `apps.domains.clinic.services.lifecycle` is now the canonical write boundary for participant creation, student booking changes, status transitions, completion, and completion cancellation.
- `create_participant()` owns tenant/student/enrollment validation, student self-booking policy, duplicate checks, capacity checks, enrollment auto-resolution, and reservation notification context.
- `change_participant_booking()` owns student-only pending-booking changes, malformed/same-session guards, row-locked destination capacity checks, duplicate prevention, cancel-after-new-booking atomicity, and `clinic_reservation_changed` notification context.
- `ParticipantViewSet.create()` and `change_booking()` are now transport facades over the service layer and only handle serializer output plus `transaction.on_commit()` dispatch.
- Cancel/change notification variables for existing schedule, changed schedule, and modifier now come from the service boundary instead of the HTTP view.

Compatibility boundary:

- AlimTalk dispatch still uses the existing `transaction.on_commit()` compatibility path. Event/outbox extraction remains the next clinic/messaging durability slice.

## 2. Student Clinic Change UX Contract

- Production QA showed tenant 1 auto-approves student clinic bookings: a student-created booking becomes `booked`, while the backend correctly permits `change-booking` only for `pending`.
- The student clinic page now mirrors that contract:
  - `pending` bookings show the existing change flow.
  - `booked`/`approved` bookings show "예약 확인", lock the session buttons, hide "일정 변경 신청하기", and guide the user to contact the academy.
  - `handleChangeRequest()` also hard-guards non-pending bookings so stale UI state cannot call the backend mutation.
