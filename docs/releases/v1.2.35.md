# V1.2.35 Release Notes

**Status:** production-deployed pending verification
**Release date:** 2026-06-06 KST
**Scope:** Common Alimtalk SSOT + account password recovery delivery hardening

This release closes the messaging policy gap where password/account notices and
manual/automatic notifications could still be reasoned about as SMS,
tenant-specific Alimtalk, or template-fallback sends.

## 1. Root Cause

- The product decision is common owner Alimtalk only, but legacy code still
  preserved SMS mode branches, tenant PFID/provider wording, and template
  fallback behavior.
- Password reset/account recovery had multiple entrypoints. Some paths could
  reset a password before delivery was proven, and legacy OTP endpoints still
  existed beside the canonical account recovery flow.
- Notification logs did not carry enough account target metadata to verify, from
  the student detail or operations side, which student/account a common-owner
  send belonged to.
- Worker safety needed to hold even for raw or old SQS payloads, not only for
  the current API enqueue path.

## 2. Fix Summary

- SMS/LMS real sending is disabled service-wide.
- All new Alimtalk enqueue calls normalize `tenant_id` to `OWNER_TENANT_ID`; the
  original business tenant is recorded as `source_tenant_id`.
- The messaging worker re-normalizes raw/legacy SQS payloads to the common owner
  tenant and uses `source_tenant_id` only for business checks such as reservation
  cancellation.
- Tenant-specific PFID/provider/self credentials are ignored for real sends.
- Template fallback is removed. Exact owner-approved templates or explicitly
  mapped unified templates are required; missing approval fails closed.
- Account/password recovery sends through `send_alimtalk_via_owner()` with
  account target metadata and rollback on delivery failure.
- Legacy OTP password-find endpoints now return 410 and do not mutate cache,
  password, or delivery state.
- `NotificationLog` now stores `source_tenant_id`, `target_type`, `target_id`,
  and `target_name` for account-notification traceability.
- Messaging/account domain docs and SSOT rules now state common-owner Alimtalk
  only, no SMS, no tenant channel, and no fallback.
