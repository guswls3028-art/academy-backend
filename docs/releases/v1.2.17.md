# V1.2.17 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-01 KST
**Scope:** OMR mixed objective/essay sheet shape and tenant 2 real grading verification

This release fixes the recognition-shape defect behind the tenant 2 report that
OMR marking recognition still looked wrong after the readiness architecture
release. The affected sheet had 20 objective questions plus 5 essay questions,
but the legacy OMR payload treated `Sheet.total_questions=25` as 25 objective
marking slots. That made the detector split the objective area into the wrong
columns and read questions 14-20 from the essay/blank area.

## 1. Root Cause

- Tenant 2 exam 333 used a mixed sheet:
  - 20 objective OMR questions.
  - 5 essay questions.
  - `Sheet.total_questions=25`.
- The worker payload used `total_questions` as the objective count.
- `build_omr_meta(question_count=25)` created a two-column objective layout for
  25 choices, while the real scan only had one objective column for questions
  1-20.
- Questions 14-20 were therefore read from the right-side essay region instead
  of the actual objective marks.

## 2. Fix

- Added explicit OMR shape fields to `Sheet`:
  - `choice_count`
  - `essay_count`
- Added migration `exams.0017_sheet_omr_shape`.
- Added canonical sheet-shape resolution in `apps.support.omr.sheet_shape`.
- OMR payload generation now sends the objective count separately from the
  total question count.
- Legacy sheets without explicit shape infer objective/essay split from
  `AnswerKey` values:
  - numeric `1`-`5` or comma-separated numeric values remain objective,
  - the first non-choice answer starts the essay segment.
- OMR PDF/template generation and `save_as_template` now preserve the explicit
  shape instead of collapsing everything back into one total.

## 3. Tests

- Added regression coverage for legacy answer-key inference on mixed

- Added template-copy coverage so derived sheets keep OMR shape fields.
