# V1.2.6 Release Notes

**Status:** production-deployed; final seal pending optional device-body content validation  
**Release date:** 2026-05-23 KST  
**Scope:** Excel student import orchestration SSOT + teacher import UX + production hardening

## 1. Excel Import SSOT

- Added `apps.domains.students.services.import_students`.
- Student-only Excel import and lecture-enroll Excel now share:
  tenant-scoped name+parent duplicate detection, deleted-student restore, school-level validation, phone/ps_number/omr derivation, and `create_student_account()`.
- `ExcelParsingService` calls `import_students_from_rows()` for student-only imports.
- `lecture_enroll_from_excel_rows()` calls `resolve_student_import_row()` for imported students.
- `bulk_from_excel.py` and `lecture_enroll.py` remain compatibility facades.
- Welcome AlimTalk is sent only for newly created students when `send_welcome_message=true`.

## 2. Teacher Import UX

- Teacher mobile Excel import no longer uploads immediately with hidden `0000`.
- File selection opens an Excel import bottom sheet.
- The sheet explicitly confirms initial password and welcome AlimTalk flag before upload.
- `src/app_teacher/domains/students/api.ts` forwards `send_welcome_message` to the shared student upload contract.

## 3. Production QA Findings And Fixes

- Production QA exposed `/api/v1/students/excel_job_status/<job_id>/` returning 404.
  Root cause: DRF `@action(url_path=...)` used a Django path converter string instead of a regex capture.
  Fix: `StudentViewSet.excel_job_status` now uses `(?P<job_id>[^/.]+)` and has a route regression test.
- Production AlimTalk-on QA exposed a parser heuristic bug:
  names longer than 20 characters were discarded before valid-phone evidence was considered.
  Fix: valid phone rows now allow names up to the model limit of 50 characters; long no-phone rows are still filtered as likely headings.
- The first failed long-name AlimTalk QA created no student residue.
