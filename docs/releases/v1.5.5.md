# V1.5.5 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-06-29 KST
**Scope:** Batched matchup hit-report candidate loading

This patch closes the hit-report editor UX debt where large exam documents could
show a long blocking loading state while the server searched every candidate in
one request.

## 1. Decision

GO for production use.

The editor now opens from the quick draft first, then loads candidate matches in
bounded batches. The reported "keeps loading" class is closed for the verified
document `976` workflow: the modal renders in about two seconds, existing 1:N
selected materials are immediately visible, and candidate rows fill in without a
blocking full-screen spinner.

## 2. Change Summary

- Added backend `mode=candidates&problem_ids=...` support for hit-report draft
  candidate batches, capped to 32 exam-problem ids per request.
- Kept the existing quick draft and full draft contracts compatible.
- Changed the frontend editor to load the quick draft first, then fetch
  candidates in 16-problem batches and merge only the returned candidates into
  the existing report state.
- Removed the slow full-draft fallback from the initial editor load so a
  candidate-search timeout cannot leave the user stuck behind the initial
  blocking spinner.
- Updated active-candidate selection so newly arrived batched candidates appear
  in the preview naturally.

## 5. Residual Risk

- Candidate search for large documents is still computationally heavy; the
  total background time for 83 questions was about 100s. The user is no longer
  blocked, but a future optimization could cache candidate batches or precompute
  report candidates after analysis.
- The legacy full draft endpoint remains available for compatibility and can
  still be slow on large documents. The production editor no longer depends on
  that path for initial loading.
