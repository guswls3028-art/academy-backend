# V1.8.0 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-07-14 KST
**Scope:** Enterprise stabilization: tenant isolation, state-machine integrity,
immutable deployment control, production-data constraints, and safe real-use QA

This release closes a whole-product stability audit across the Django/DRF
backend, React frontend, workers, AWS deployment control plane, and production
data. It repairs verified inconsistencies, moves important business invariants
into transactional writers and database constraints, and makes production E2E
safe enough to remain a blocking deployment gate.

## 1. Decision

GO for production use.

The final backend suite, frontend quality gate, database migrations, production
data canary, runtime-image verification, tenant availability checks, and
real-use round trips all passed. No Critical or High finding remains in the
release-scoped surface.

## 2. Product and Domain Stabilization

- Enforced fail-closed student video and clinic access resolution, including
  same-tenant enrollment ownership and malformed/unowned child selections.
- Added tenant-safe owner messaging logs and quota accounting.
- Hardened billing transitions for PROCESSING/ambiguous provider outcomes,
  partial refunds, webhook reconciliation, scheduled cancellation restoration,
  monotonic subscription periods, and stale active/grace lifecycle states.
- Encrypted Toss billing keys with a dedicated Fernet keyring and rotation
  workflow instead of storing provider credentials as plaintext or coupling
  them to Django `SECRET_KEY`.
- Introduced durable messaging delivery states, atomic claiming/retry rules,
  occurrence-key idempotency, sensitive terminal-log redaction, preview-token
  retention, and migration preflight for malformed legacy payloads.
- Serialized payroll/work-record, monthly-close, messaging-quota, billing
  profile, and video ordering writers against concurrent updates.
- Made secession terminal, fee-payment inputs, notification occurrences, and
  video/folder ordering explicit business invariants.
- Clarified subscription price/state presentation across owner, teacher, and
  developer surfaces and improved auth-unavailable recovery, mobile staff
  management, exam bundle management, video ordering/playback, and messaging
  feedback.

## 3. Security and Deployment Control

- Replaced process-local login throttling with a database-backed fixed window
  and derived client IP only from trusted proxy chains.
- Added tenant/auth regression suites for JWT claims, middleware bypasses,
  parent-child selection, student video access, and cross-tenant writes.
- Upgraded `cryptography` to `48.0.1`; `pip-audit` reports zero known
  vulnerabilities for the production Python dependency set. Frontend package
  audit also reports zero known vulnerabilities.
- Made the six ECR repositories immutable except for the compatibility
  `latest` tag, pinned ASG/Batch runtime selection to verified digests, and
  promoted a complete release manifest only after successful verification.
- Added one DynamoDB-backed deployment/data-mutation lock shared by CI,
  deploy, migration, cleanup, repair, and manual production maintenance paths.
- Restricted legacy pruning and IAM permissions to canonical, explicitly
  named resources and made worker deployment depend on the migration gate.
- Created an encrypted, versioned, public-blocked operations backup bucket with
  lifecycle retention and took a pre-repair RDS snapshot.
- Separated CI from the real tenant owner by provisioning a dedicated
  Enterprise QA account; its credential is held only in GitHub Actions secrets
  and was transported through a temporary encrypted SSM parameter that was
  deleted immediately after rotation.

## 10. Intentional Deferrals

  contract secrets are not configured. The canary explicitly enforces that
  this cannot be enabled unsafely.
- The `sswe` monthly price remains the observed manual/promotional `150,000`
  KRW rather than being silently overwritten with the `198,000` KRW default;
  changing that commercial agreement requires product-owner evidence.
- Django emits `security.W008` inside the container because TLS redirect is
  enforced at the ALB instead of Django. The public HTTP-to-HTTPS redirect and
  HTTPS health paths passed; no duplicate application redirect was introduced.
