# V1.4.4 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-06-22 KST
**Scope:** Submission lifecycle transition SSOT + deploy regression guard

This patch closes the submission state-machine drift found during the project
review. The runtime write paths now converge on a public lifecycle facade, the
`EXTRACTING` state is part of the allowed flow, recovery/cascade paths pass
through the same transition guard, and CI blocks future direct runtime imports
of the low-level transition engine.

## 1. Decision

GO for current production operation.

The submission state transition risk is no longer a docs-only concern. The
state machine is exercised by service tests, runtime callers use the lifecycle
API, and the deploy lint gate rejects new direct `transition.py` runtime
imports before build/deploy.

## 2. Change Summary

Backend:

- Added `EXTRACTING` to the executable `STATUS_FLOW` and aligned the domain
  state-transition documentation with the implementation.
- Introduced `apps/domains/submissions/services/lifecycle.py` as the public
  submission lifecycle facade over the low-level transition engine.
- Moved cross-domain submission/student-app dependencies into support-layer
  dependency modules so views/services do not reach through internal domain
  boundaries directly.
- Reworked submission, OMR, result recalculation, student exam, state recovery,
  and supersede paths to use meaningful lifecycle operations instead of raw
  status writes.
- Added regression tests for lifecycle transitions, EXTRACTING recovery, stuck
  recovery, idempotency, and forbidden terminal-state rewrites.
- Added `scripts/lint/check_submission_lifecycle_boundary.py` and wired it into
  the backend deploy workflow after Ruff, before refactor-boundary guards and
  smoke tests.
- Refreshed production deploy evidence reports after the release deployment.

Frontend:

- Not changed.
