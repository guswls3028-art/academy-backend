# V1.2.16 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-01 KST
**Scope:** OMR fact/readiness architecture and zero-answer grading prevention

This release moves the fragile OMR completion path away from a single
`Submission.status` interpretation. OMR now records durable recognition,
detected-answer, and student-match facts, and all automatic grading goes through
a readiness gate. The goal is to stop the repeated class where a teacher manually
matches a student while AI recognition is still incomplete and the system
prematurely publishes a zero-answer score.

## 1. OMR Facts

- Added `OMRRecognitionRun` to persist each AI recognition callback, worker
  version, alignment summary, answer counts, status counts, raw result, and job
  id.
- Added `OMRDetectedAnswer` to preserve question-level detected markings from
  each recognition run, separate from the editable `SubmissionAnswer`
  compatibility projection.
- Added `OMRStudentMatch` to preserve manual, auto, duplicate, and unmatched
  student-match history with a single current match per submission.
- Added migration
  `submissions.0005_omrrecognitionrun_omrdetectedanswer_omrstudentmatch_and_more`.

## 2. Readiness Gate

- Added `academy.application.use_cases.omr.grading_readiness`.
- The canonical gate now requires:
  - an OMR exam submission,
  - a confirmed/current student match or legacy enrollment,
  - at least one persisted answer,
  - and a non-terminal/non-failed state that can safely transition to grading.
- Manual student matching now records a student-match fact and returns
  `graded=false` with `readiness.missing=["answers"]` when no answers exist.
- AI callbacks, manual edit, duplicate selection, and late-answer recovery now
  use the same readiness gate before grading.

## 3. Manual Match + Late AI

- If a teacher selects the student first and AI answers arrive later with a
  blank/unreadable identifier, the mapper preserves the manual match after
  revalidating exam membership and duplicate conflicts.
- A later AI answer payload can then hydrate `SubmissionAnswer`, record
  recognition facts, transition through `answers_ready`, and grade normally.
- Duplicate or unmatched identification facts block grading instead of falling
  back to a stale `submission.enrollment_id`.

## 4. Tests

- Added a regression proving manual student match with no answers does not
  grade or create a zero-score `Result`.
- Added a regression proving late AI answers after a manual match preserve the
  selected student, record recognition facts, and grade to the expected score.
- Added assertions that automatic identifier mapping records recognition,
  detected-answer, and current student-match facts.
- Updated AI callback unit tests to assert the readiness gate rather than the
  old direct grading task.

## 6. Notes

- This release fixes the OMR state/order architecture and premature zero-score
  failure class.
- It does not claim to solve raw scan marking accuracy by itself. The new fact
  tables make recognition outputs auditable so real scan recognition tuning can
  be handled with evidence instead of guessing from the final score alone.
- No frontend change.
