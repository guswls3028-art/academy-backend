# V1.2.49 Release Notes

**Status:** Production-deployed and launch GO verified; payment excluded
**Release date:** 2026-06-11 KST
**Scope:** Student-domain launch stability, video upload/HLS integrity, all-menu audit, API launch baseline

This release seals the student-domain stabilization pass for controlled expansion.
The payment path remains outside this release and must pass a separate payment
gate before any billing or settlement promotion.

## 1. Decision

GO for controlled expansion, excluding real payment, refund, settlement, billing
charge, and card mutation paths.

The release is considered production-proven for the verified student/account,
content, video, worker, and infrastructure paths listed below. Remaining risk is
explicitly bounded in section 5.

## 2. Change Summary

Frontend:

- Added transient read recovery for idempotent requests only, so brief backend or
  network faults do not strand student/staff read screens.
- Hardened the all-menu/all-button Playwright audit against transient page-read
  defects without hiding reproducible runtime failures.
- Updated the student mobile drawer selector used by the E2E audit.
- Confirmed current frontend HEAD `900bb32d` quality gate success after the
  answer-key modal lock fix.

Backend:

- Restored proctored student playback by replacing removed
  `django.utils.timezone.utc` usage with `datetime.timezone.utc`.
- Made HLS local validation and R2 upload integrity use the same short-video
  segment policy, preventing valid short clips from failing in the worker.
- Kept video thumbnail and non-empty segment checks as hard integrity gates.
- Aligned exam question tenant filtering with the sheet/exam tenant relation and
  kept template editability in its SSOT helper.

Infrastructure and operations:

- Set API launch baseline to ASG min/desired/max `2/2/3` in workflow, SSOT,
  deployment verification, and cleanup scripts.
- Fixed Windows ECR lifecycle policy application by passing compressed JSON
  directly.
- Updated deploy-worker verification so non-runtime commits after a successful
  image build are classified correctly instead of producing false image drift.
