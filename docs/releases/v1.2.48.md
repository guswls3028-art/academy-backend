# V1.2.48 Release Notes

**Status:** Production-deployed and tenant maintenance-lock recurrence blocked  
**Release date:** 2026-06-10 KST  
**Scope:** Global maintenance toggle incident fix and prevention

This release resolves the tenant screen lock incident where non-platform tenant
domains were redirected to `/maintenance` after the global maintenance flag was
enabled from the developer dashboard.

## 1. Decision

Global maintenance ON is no longer an operable production control.

The backend is the hard gate: `PATCH /api/v1/core/maintenance-mode/` rejects
`enabled=true` with `global_maintenance_disabled`. The frontend developer
dashboard no longer renders the ON switch, but the server-side rejection is the
source of truth.

OFF remains available as an incident-recovery operation and removes only the
`maintenance_mode` feature flag key while preserving all other feature flags.

## 2. Change Summary

Backend:

- Blocked global maintenance ON in `MaintenanceModeView.patch`.
- Kept maintenance OFF as a safe key-removal recovery path.
- Made maintenance count/removal logic portable across test and production DB
  engines.
- Added regression tests for blocked ON and safe OFF.

Frontend:

- Removed the developer dashboard global maintenance switch.
- Kept a read-only maintenance status card.

Incident documentation:

- Added `docs/reports/incidents/incident-2026-06-10-global-maintenance-toggle.md`.

## 4. Residual Risk

- The recurrence path that globally locks every non-platform tenant from the
  developer dashboard is blocked server-side.
- Tenant pages can still show unrelated business-state blocks, such as

  `sswe` were not in maintenance mode but did show existing subscription-expired
  state.
