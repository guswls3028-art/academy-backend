# V1.2.12 Release Notes

**Status:** production-deployed
**Release date:** 2026-05-30 KST
**Scope:** OMR same-student duplicate cluster UX + OMR pipeline domain split (Phases A–G) + automatic state recovery cron

This release closes the tchul (박철과학) "OMR 이상해요" incident class and
the underlying code-debt that made every prior fix patch the same three files
in a row.

## 1. Same-Student Duplicate Cluster

- When a single student has more than one live OMR scan for the same exam,
  the review workspace now lists every sibling scan inline (thumbnail +
  status + time). Reviewers compare side by side and confirm one with
  "이 답안 채택"; the backend transitions the chosen submission to DONE,
  marks each DONE sibling SUPERSEDED, and discards in-flight siblings as
  `discarded:duplicate` in a single transaction.
- Backend endpoint `POST /submissions/<id>/accept-from-duplicates/`
  centralises the cleanup so the UI no longer has to call discard-batch
  + manual-edit by hand.
- The cluster panel reads from `manual_edit` GET, which now returns a
  `duplicate_siblings` array. No new field is required on UI components
  outside the OMR review workspace.

## 2. OMR Pipeline Domain Split

The `apply_omr_ai_result` orchestrator was 460 lines mixing worker-payload
parsing, identifier matching, enrollment locking, answer persistence, and
state transitions. Every recent recognition fix had to thread through all
of those concerns. Phases A–G split each concern into its own module:

- **Phase A — worker contract.** `apps/domains/submissions/omr_pipeline/contracts/worker_response.py` is now the SSOT for what a worker callback may look like. Pydantic v2 model with `SUPPORTED_WORKER_VERSIONS` (frozen) + `parse_worker_callback()` that normalises legacy "result body at top level" payloads. Schema violation flips `manual_review.required` + `meta.worker_contract_violation` + logs `OMR_WORKER_CONTRACT_VIOLATION` so silent failures stop reaching the operator screen as blank sheets.
- **Phase B — identifier matcher.** `omr_pipeline/services/identifier_matcher.py` owns the exact / fuzzy / ambiguous decision and caches the per-exam enrollment tail index once per batch. New 1-digit silent error defence: even when the worker reports `status='ok'`, if any digit is `ambiguous` the matcher checks whether a 1-digit variant points to a different student in the same exam. Pure `ok` matches still trust the worker, so the false positive rate stays at 0 against the audited tchul history.
- **Phase C — answer persister.** `omr_pipeline/services/answer_persister.py` owns the question-id mapping, `SubmissionAnswer` upserts, `answer_stats` aggregation, and the ANSWER_* / `ALIGNMENT_FAILED` / `ANSWER_QNUM_NOT_IN_SHEET` review reason rules. `apply_omr_ai_result` calls it once and reads an `AnswerPersistResult` dataclass.
- **Phase D — state recovery.** `omr_pipeline/services/state_recovery.py` and the `recover_stuck_omr_submissions` management command auto-fail SUBMITTED / DISPATCHED / EXTRACTING / GRADING submissions older than 30 minutes. The transition is race-safe (re-selects with `select_for_update`), records `meta.state_recovery`, and emits `OMR_STATE_RECOVERY` log events.
- **Phase E — enrollment finalizer.** `omr_pipeline/services/enrollment_finalizer.py` runs IdentifierMatcher + `lock_exam_enrollment_candidate` + `find_conflicting_exam_submission` as a single call, returning an `EnrollmentFinalizeResult`. The mapper just unpacks it.
- **Phase F — dispatcher split.** `apps/support/omr/sheet_resolver.py` and `apps/support/omr/payload_builder.py` own OMR-specific Sheet scoping and worker payload construction. `apps/domains/submissions/services/dispatcher.py` is now a thin orchestrator (246 → 176 lines) that delegates per source and re-exports the resolver functions so existing call sites keep working.
- **Phase G — `extra='forbid'`.** Audited 30 days of prod `meta.ai_result` keys on all 88 OMR submissions: every top-level key sits inside `_ENVELOPE_KEYS`. Flipped `OMRWorkerCallback.model_config` to `extra='forbid'` so any future worker field needs an explicit schema + backend deploy. Normalisation still folds legacy "result body inlined at top" payloads into `result`, so existing producers are unaffected.

`apply_omr_ai_result` shrank from 460 to 220 lines; `dispatcher.py` from
246 to 176. The OMR review UI workspace continues to work unchanged
against the now-thin orchestrator.

## 3. Automatic State Recovery Operations

- New AWS EventBridge rule `academy-v1-recover-stuck-omr` (rate(5m)) calls
  the `recover_stuck_omr_submissions` management command on the
  `academy-v1-api` ASG via SSM `AWS-RunShellScript`.
- New IAM role `academy-v1-eventbridge-ssm-omr-recovery-role` with policy
  `academy-v1-eventbridge-ssm-omr-recovery-policy` (Allow:
  `ssm:SendCommand` on `AWS-RunShellScript` + EC2 instances in
  ap-northeast-2).
- The rule was registered after Phase D's backend deploy went green, so
  the very first fire ran against the new code.

## 7. Residual Observations

- Phase A `extra='ignore'` → Phase G `extra='forbid'` flip relies on
  the prod audit window (30 days, 88 rows). Any future worker version
  that ships a new envelope key must update
  `SUPPORTED_WORKER_VERSIONS` and the `OMRWorkerCallback` schema; a
  silent prod-only key will surface as
  `OMR_WORKER_CONTRACT_VIOLATION` on the first callback.
- The EventBridge recovery rule targets by `tag:Name=academy-v1-api`.
  An ASG refresh that briefly leaves zero tagged instances will skip
  one or two ticks; the next 5-minute fire picks back up. The 30-minute
  recovery timeout absorbs that gap.
- `dispatcher.py` still owns ONLINE / HOMEWORK paths in addition to
  the thin OMR delegate. Splitting Homework into its own pipeline is
  intentionally left as a future initiative — the OMR domain split is
  already large enough to ship and audit independently.
- The "박철과학 회신" message text drafted alongside this work is a
  user-side decision and not part of the deployed surface.
