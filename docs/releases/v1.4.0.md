# V1.4.0 Release Notes

**Status:** Production-deployed / GO  
**Release date:** 2026-06-14 KST  
**Scope:** API edge HTTP-to-HTTPS enforcement, deploy verification realignment

This release closes the `security.W008` investigation. The warning was not
caused by a missing secure-cookie or HSTS setting. It was caused by the public
ALB port 80 listener forwarding plain HTTP traffic to Django while production
settings intentionally keep `SECURE_SSL_REDIRECT=False` to avoid breaking
private/internal HTTP worker paths.

## 1. Decision

GO for current production operation.

The production ALB port 80 listener now redirects to HTTPS:443, public
`https://api.hakwonplus.com/healthz` is healthy, and the PostDeploy canary plus
deploy verification both pass.

## 2. Change Summary

Infrastructure:

- Changed the live `academy-v1-api-alb` HTTP:80 listener default action from
  target-group forward to HTTPS:443 redirect.
- Updated `scripts/v1/resources/alb.ps1` so future deploys preserve the HTTP
  redirect instead of recreating the old plain-HTTP forward behavior.
- Confirmed the API ASG instance refresh triggered during idempotency
  verification completed successfully.

Operations:

- Added `api_http_redirects_to_https` to `run-production-canary.ps1`.
- Updated `run-deploy-verification.ps1` to verify ALB HTTP redirect plus public
  HTTPS API health, rather than following the ALB DNS redirect to a certificate
  mismatched hostname.
- Updated API edge architecture and production canary runbook docs.

## 5. Residual Risk

- `django check --deploy` still emits `security.W008` because Django only sees
  `SECURE_SSL_REDIRECT=False`; it cannot inspect the ALB listener. This is now
  an expected app-layer warning with an infra-layer control: ALB HTTP:80
  redirects to HTTPS and the canary fails if plain HTTP reaches Django again.
- Django `SECURE_SSL_REDIRECT` remains off intentionally so private/internal
  HTTP worker paths are not redirected to invalid HTTPS private hosts.
