# V1.2.3 Release Notes

**Status:** production-deployed; final seal pending device-body validation  
**Release date:** 2026-05-23 KST  
**Scope:** student lifecycle convergence, restore skip UX, account recovery temporary password usability

## 1. Student Lifecycle

- Student permanent deletion now routes through `students.services.permanently_delete_students`.
- `bulk_permanent_delete`, conflict re-register deletion, deleted-duplicate cleanup, and 30-day purge share tenant-scoped locking and cleanup behavior.
- The canonical delete path removes pending password resets, fee/section dependencies, learning artifacts, media/progress references, memberships, and orphan users with retained-account guards.
- Soft delete no longer globally deactivates a user account when that user is still shared by protected same-tenant roles or cross-tenant records.

Remaining structural debt: the permanent-delete service still owns a guarded cross-domain SQL cleanup graph. The next durable boundary is domain-owned cleanup hooks/events, not new view-level deletion logic.

## 2. Restore UX

- Frontend student restore now treats backend `skipped` rows as first-class contract data.
- Deleted-student bulk restore warns the operator when a student cannot be restored because an active `ps_number` already exists.
- Create-conflict restore modal no longer reports success when backend restored count is zero.

## 3. Account Recovery

- Account recovery temporary passwords are now 6 numeric digits.
- SSOT: `apps/core/services/password.py` (`TEMP_PASSWORD_LENGTH = 6`).
- Public login recovery remains on `/api/v1/auth/account-recovery/dispatch/`.
- Password recovery still uses pending reset semantics: the old password stays valid until the delivered temporary password is used at login.
- Account/credential notification bodies remain redacted in `NotificationLog.message_body`.
