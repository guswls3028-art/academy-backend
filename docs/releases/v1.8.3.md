## 2. Student and Parent Product Contract

- `GET /api/v1/student/grades/` now returns the same canonical `exam_trend` and
  `exam_summary` progression contract used by staff views. A newly scored
  regular exam appears on the next read without a batch job, cache rebuild, or
  manual configuration.
- Trend points are chronological rounds across distinct eligible exams. Scores
  with different maxima are normalized to percentage points while preserving
  raw score/max-score tooltips and bonus scores above 100%.
- Zero remains a valid score. Non-finite, negative, or non-positive-maximum
  values do not enter analytics. `NOT_SUBMITTED` stays visible as `미응시` but
  is excluded from trend, averages, and pass-rate denominators.
- The screen shows latest, average, and best results plus previous-round and
  first-to-latest changes. Long histories scroll horizontally and keep readable
  round labels instead of compressing the chart.
- Empty, loading, and independent analytics-failure states are explicit and do
  not present missing data as a real zero-score history.

## 3. Parent/Student Experience

- The student 성적 screen uses the shared responsive score-trend chart with
  polished mobile, desktop, and dark-mode presentation.
- Parent accounts see the selected child's exact progression. Switching from
  child A to child B clears all student-scoped query caches, waits for selection
  initialization, and sends the exact `X-Student-Id` header before rendering
  child-specific content.
- Parent children are limited to active same-tenant enrollments returned by
  `/core/me`; cross-tenant or inactive relationships fail closed.
- Responsive QA covered 390px student, 390px parent child-switch, dark mode,
  tooltip and comparison states, analytics-only failure, and a 14-round history
  with no viewport overflow.

## 4. Tenant and Business-Logic Hardening

- Student grade history resolves student, enrollment, exam, session, lecture,
  homework, template, clinic, and attempt metadata through the request tenant.
- Representative attempts must match the exact enrollment/exam pair. Foreign
  or corrupt attempts, templates, sessions, structure IDs, and answer metadata
  cannot leak into the response or inflate counts.
- Legacy same-tenant template questions remain supported; a template from a
  different tenant fails closed. Question and answer-key visibility also uses
  the effective exam and tenant scope.
- Each distinct exam contributes at most one trend point using the latest valid
  representative result. Archived history remains available; system lectures
  and non-submissions do not distort analytics.
- Achievement and ranking calculations use the same exact-attempt contract.
  Homework history uses the assignment timestamp, and analytics apply one
  consistent date window across exam and homework facts.

## 5. Delivery and Cleanup Hardening

- Frontend build identity is derived from the platform commit SHA, making the
  GitHub build and Cloudflare build byte-for-byte deterministic. Two same-SHA
  builds produced identical SHA-256 hashes for all 700 output files.
- The deployment gate verifies the exact version plus canonical, query-less
  critical chunks for admin, student, dashboard, clinic, community, lecture,
  and score-entry routes before any canary runs. The probe is bounded by a
  wall-clock deadline and requires consecutive stable reads.
- Missing static assets return a typed 404 with `Cache-Control: no-store,
  no-cache, must-revalidate`; redirects and 304 responses keep their native
  semantics. The final production probe returned 404 with `CF-Cache-Status:
  BYPASS`.
- QnA production E2E cleanup now deletes with the author student account and
  treats a non-200/204 cleanup response as a blocking failure. Two explicit
  `[E2E]` rows found by the first post-deploy canary were dry-run scoped,
  transactionally removed, and the final workflow plus canary confirmed zero
  residue.
