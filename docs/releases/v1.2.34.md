# V1.2.34 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-05 KST
**Scope:** OMR AI worker scale lifecycle stabilization + production canary proof

This release closes the recurring OMR "uploaded but not recognized" incident by
moving the AI worker scale-in decision to the worker itself and proving the
full production path with tenant 1 real-use canaries.

## 1. Root Cause

- The OMR upload API created AI jobs and SQS messages, but the API EC2 role did
  not have the scoped Auto Scaling permissions needed to wake
  `academy-v1-ai-worker-asg` from desired capacity `0`.
- After wake permission was added, the legacy `ai-worker-queue-low` CloudWatch
  alarm could still run scale-in from lagging SQS metrics. In practice, it could
  return desired capacity to `0` before the fresh worker finished pulling the
  image and consuming the queued OMR job.
- The old lifecycle split was structurally fragile:
  - API/CloudWatch could scale out;
  - CloudWatch could scale in from stale queue metrics;
  - the worker did not own the final "queue is truly idle" decision.
- During deployment, Video Ops Batch SSOT also had an unsupported instance type
  (`t4g.medium`), which blocked `deploy.ps1` while recreating the Ops compute
  environment.

## 2. Fix Summary

- Added a scoped API EC2 inline policy that allows:
  - `autoscaling:DescribeAutoScalingGroups`;
  - `autoscaling:SetDesiredCapacity` only for `academy-v1-ai-worker-asg`;
  - EC2 start/describe actions only for tagged academy AI worker resources.
- Added queue count reads to the SQS queue adapter/port so runtime code can
  inspect live visible, in-flight, and delayed message counts.
- Changed AI worker idle scale-in to worker-owned logic:
  - worker polls live SQS attributes;
  - scale-in happens only after repeated empty polls;
  - queue-count failure is treated as non-idle.
- Converted `ai-worker-queue-low` to observability-only:
  - alarm actions disabled;
  - legacy `ai-worker-scale-in` policy removed.
- Updated deployment SSOT to use supported Video Ops Batch instance type
  `c6g.medium`, allowing the production deploy path to complete cleanly.
