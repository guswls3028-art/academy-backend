# V1.6.5 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-07-08 KST
**Scope:** YouTube link video upload + student playback parity + playlist numeric ordering

This patch release lets teachers add YouTube links next to existing uploaded
videos, and lets students watch both source types from the same video learning
surface without confusing controls or list semantics.

## 1. Decision

GO for production use.

Backend and frontend `main` are deployed. Teachers can create a READY video
from a YouTube URL without Batch transcoding. Student playback returns a
YouTube embed payload, the player uses the same page shell and playlist as
uploaded videos, and YouTube-native controls are exposed for provider playback.
Production canary, deploy verification, uploaded-video CDN chain smoke, and a
real production YouTube create/play/delete smoke all passed.

## 2. Change Summary

- Added `source_type`, `youtube_video_id`, and `youtube_url` to video records,
  with migration `video.0018_video_source_type_youtube`.
- Added `POST /api/v1/media/videos/youtube/` for teacher/admin YouTube link
  creation. YouTube videos are immediately `READY` and do not enter the HLS
  transcode pipeline.
- Added shared YouTube URL parsing, thumbnail, watch URL, and embed URL helpers.
- Extended admin upload modal with `file` and `YouTube link` modes, link
  parsing, thumbnail preview, and the same playback policy controls.
- Extended admin and student video lists with YouTube source badges so uploaded
  and linked videos can be mixed without ambiguity.
- Extended student playback payloads and player code for YouTube source videos.
  The app keeps the same title, sidebar, playlist, comments, and learning-state
  shell, while the iframe exposes YouTube-native controls for actual playback.
- Fixed playlist ordering so titles sharing the same base name and numeric
  suffix sort by suffix number, for example `영상이름 - 1`, `영상이름 - 2`,
  `영상이름 - 3`, independent of encoding completion timing.
- Updated video operations checks and production canary so READY YouTube links
  are not treated as HLS-transcoded videos.
- Normalized Windows-to-SSM command newlines in the deploy verification helper.

## 5. Audit Lenses

- Backend domain audit: YouTube source videos use explicit `source_type` and
  canonical IDs instead of overloading HLS/file fields; HLS-only invariants now
  exclude YouTube by source type.
- Tenant isolation audit: YouTube creation still resolves the session through
  the lecture tenant and rejects cross-tenant uploads.
- Frontend integration audit: admin creation, student list, playback payload,
  and player source detection all share the same source-type contract.
- Destructive test audit: malformed YouTube URLs fail before create; test
  production rows were deleted and residue was rechecked after every live smoke.
- UX/accessibility audit: file upload remains the default tab; YouTube links
  get preview and badges; student playback keeps one learning shell while using
  provider-native playback controls where that is the reliable interaction.
- Security audit: no auth bypass, unsigned media URL fallback, or tenant
  fallback path was introduced; YouTube embed URLs are derived from parsed video
  IDs rather than arbitrary iframe HTML.

## 6. Residual Risk

- YouTube availability depends on the external provider. If a video owner
  disables embedding or removes the video, the item remains a valid link record
  but playback will show the provider failure state.
- YouTube iframe internals are cross-origin, so the app cannot fully duplicate
  every custom HLS control. The chosen production UX keeps the app shell
  consistent and lets YouTube own play/pause/fullscreen inside the iframe.
