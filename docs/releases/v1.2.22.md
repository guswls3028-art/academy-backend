# V1.2.22 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-02 KST
**Scope:** OMR objective-only decorative essay area + static sheet production route proof

This release closes the remaining OMR shape issue for objective-only exams. An
exam can now remain logically objective-only while the printed/static OMR sheet
still shows a 5-row essay-looking space for visual balance. The decorative area
does not participate in recognition, answer payloads, scoring, result totals, or
teacher answer-key contracts.

## 1. Root Cause Themes

- Teachers sometimes added 5 essay rows only because a pure objective OMR sheet
  looked too empty. That made the visual layout and the grading contract mean
  different things without a clear boundary.
- The durable OMR contract needed one canonical distinction:
  `essay_count` is grading logic, while `render_essay_count` is print/display
  layout only.
- The frontend static `/omr-sheet.html` path also needed production proof
  because Cloudflare Pages normalizes `.html` routes to pretty paths.

## 2. Fix Summary

- Backend OMR document DTO now exposes:
  - logical `essay_count` for recognition/scoring contracts.
  - visual `render_essay_count` for PDF/HTML rendering.
  - decorative label `서술형 공간` when `essay_count=0` but a visual area is
    rendered.
- Backend PDF/HTML renderers use the visual render count only for sheet layout.
- OMR payload, template metadata, submission answer rows, recognition facts, and
  result scoring continue to use logical objective question count only.
- Frontend static OMR sheet now defaults to `essay=0` and renders 5 decorative
  essay rows when objective questions exist.
- Cloudflare Pages routing now lets `/omr-sheet` and `/omr-sheet.html` be served
  as static OMR HTML instead of falling through to the SPA or a redirect loop.
- The frontend quality gate now verifies both OMR static paths after deploy.
