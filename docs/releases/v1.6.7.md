# V1.6.7 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-07-09 KST
**Scope:** Ymath temporary full messaging shutdown

This patch release temporarily blocks every Alimtalk/SMS send path for Ymath
tenant id `4` until the academy owner finishes internal study and announces the
messaging rollout to users.

## 1. Decision

GO for production use.

Backend `main` is deployed. API and messaging worker images for commit
`5affed57` are in ECR and runtime verification is GO. Ymath owner-source and
direct-tenant enqueue paths return `False` before SQS/log creation, and the
messaging worker has a final fail-closed guard for queued or legacy messages.

## 2. Change Summary

- Added `TEMPORARILY_DISABLED_MESSAGING_TENANTS = {4}` to the messaging policy.
- Extended `is_messaging_disabled()` so Ymath is treated like a full messaging
  shutdown tenant, alongside the existing test-tenant guard.
- Added optional runtime extension via `MESSAGING_DISABLED_TENANT_IDS`.
- Updated `enqueue_sms()` to evaluate the business tenant, including common
  owner sends with `source_tenant_id=4`, before SQS enqueue.
- Added messaging-worker fail-closed handling so queued Ymath messages are
  deleted before any provider call.
- Added focused tests for Ymath shutdown and owner-source enqueue skip.
- Updated `docs/domain/messaging-alimtalk.md` with the temporary tenant-level
  shutdown policy.

## 5. Audit Lenses

- Backend domain audit: shutdown is centralized in the existing messaging policy
  and checked at both enqueue and worker-send boundaries.
- Tenant isolation audit: the guard uses business tenant identity, so owner
  common sends with `source_tenant_id=4` cannot bypass the shutdown.
- Frontend integration audit: no frontend contract changed; UI surfaces continue
  to call the same messaging APIs.
- Destructive test audit: production verification did not send a real message;
  it proved the no-send path by returning before SQS enqueue and log creation.
- UX/accessibility audit: not applicable; no user-visible UI changed.
- Security audit: no auth, permission, credential, or tenant lookup expansion
  was introduced.

## 6. Residual Risk

- The Ymath block is intentionally temporary. Remove tenant id `4` from
  `TEMPORARILY_DISABLED_MESSAGING_TENANTS` after the owner announces and approves
  Alimtalk use.
- No actual production Alimtalk was sent for this verification by design; the
  success criterion was that Ymath messages cannot reach SQS/provider at all.
