# V1.4.7 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-06-23 KST
**Scope:** Submission enrollment ForeignKey conversion + orphan preflight closure

This patch continues the backend ID-domain audit after v1.4.6. It converts the
next safe nullable submission reference to a Django ForeignKey, preserves the
public `enrollment_id` API contract, deploys the migration to production, and
records why several tempting remaining candidates are not safe one-step FK
conversions.

## 2. Change Summary

Backend:

- Converted nullable `Submission.enrollment_id` to a Django ForeignKey while
  preserving the deployed `enrollment_id` DB column and `_id` attname.
- Added migration normalization that sets invalid non-null submission
  enrollment refs to `NULL` before adding the FK constraint.
- Kept the DRF submission API contract explicit by preserving
  `enrollment_id` in both read/update and create serializers.
- Updated ID-domain guardrail docs and allowlist, reducing
  `check_id_domain_safety.py` from 21 to 20 warnings with 0 errors.
- Captured production read-only preflight findings for remaining candidates:
  `SubmissionAnswer.exam_question_id` has 165 invalid refs attached to missing
  target exams; `ResultFact.question_id` and `ExamAttempt.submission_id` still
  use structural `0` marker semantics.
