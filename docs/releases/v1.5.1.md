# V1.5.1 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-06-24 KST
**Scope:** Scale-to-zero return-path closure for idle workers

This patch closes the final cost-optimization edge case from v1.5.0: when an
SQS queue becomes completely idle, CloudWatch can stop publishing fresh SQS
metric datapoints. The worker scale-in alarms now treat missing idle datapoints
as breaching for scale-in only, so workers can reliably return to zero after
finishing work.

## 1. Decision

GO for current production operation.

Infra script fix commit `2a7614f59` was applied to production CloudWatch alarm
definitions. The API ASG was restored to the intended cost baseline after the
deploy refresh window. Final production canary and deploy verification both
passed with API `1/1/3` and all idle workers at `0`.

## 2. Change Summary

- Messaging worker scale-in alarm now uses `TreatMissingData=breaching`.
- Tools worker scale-in alarm now uses `TreatMissingData=breaching`.
- Messaging/tools scale-out alarms still use `TreatMissingData=notBreaching`,
  so missing queue metrics do not wake workers.
- API remote Django checks now wait briefly for the target Docker container to
  become healthy before running `docker exec`, avoiding false failures while a
  new ASG instance is still finishing user-data/container startup.
