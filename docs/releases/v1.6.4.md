# V1.6.4 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-07-08 KST
**Scope:** Alimtalk envelope enforcement + Solapi template contract + production send verification

This patch release finishes the SMS-to-Alimtalk cutover for the message domain:
manual sends, auto-send config, preview/preflight, notification dispatch, and
teacher-facing modal UX now fail closed unless an approved Kakao envelope is
available.

## 1. Decision

GO for production use.

Backend and frontend `main` are deployed. The Solapi template inventory was
queried from the live provider integration, the production messaging canary
reports zero enabled configs without approved templates, the teacher messaging
modal E2E passed on production, and a controlled Alimtalk send to
`01031217466` completed successfully.

## 2. Change Summary

- Removed SMS/freeform fallback from Alimtalk preview, preflight, send, auto
  notification dispatch, and QnA notification paths.
- Required approved Kakao envelopes for manual Alimtalk sends; unmapped teacher
  categories now force an explicit envelope choice instead of silently using an
  old SMS-style body.
- Aligned score Alimtalk variables with the provider-approved Solapi template:
  `학원이름`, `학생이름`, `강의명`, `차시명`, `선생님메모`, `사이트링크`.
- Kept exam/homework details inside `선생님메모` for score sends, matching the
  actual provider contract instead of inventing direct provider variables.
- Removed the stale `선생님메모1` alias and normalized teacher memo handling to
  `선생님메모`.
- Marked the payment notice envelope as unavailable because the previous
  provider SID is not registered in Solapi; enabled payment auto-send configs
  were disabled until a real approved template is registered.
- Updated frontend message UX so "메세지" behaves as Alimtalk: template explorer
  wording, envelope selection, preview variables, score variable construction,
  and save/use flow are all oriented around approved Kakao envelopes.
- Adjusted production canary/video validation to treat READY YouTube links as
  non-transcoded media, so post-deploy verification does not require HLS for
  YouTube source videos.
