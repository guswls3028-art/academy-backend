# V1.2.32 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-05 KST
**Scope:** Lecture session order SSOT + supplement insertion hardening

This release fixes lecture session creation and removes the old ambiguous
ordering contract where `order` had to mean both "nth regular lesson" and
"display position". The production contract is now explicit:

- regular sessions have `session_type=REGULAR` and a unique `regular_order`;
- omitted regular order means "append as n+1";
- manually provided `regular_order` creates that exact nth lesson when the
  number is available;
- supplement sessions have `session_type=SUPPLEMENT`, no `regular_order`, and
  can be placed after a selected display position;
- all read paths expose `display_label`, so supplements can sit visually between
  regular lessons without stealing an nth-lesson number.

The current main baseline also includes the already-merged legacy result
projection sync commit listed below. It was present before this seal and is
included here so the CURRENT release reflects the actual deployed main state.

## 1. Root Cause

- `Session.order` was the only persistent order field.
- Regular lessons, supplement lessons, attendance labels, progress comparisons,
  student dashboards, admin trees, teacher views, community trees, exam trees,
  video trees, and messaging labels interpreted that field in slightly different
  ways.
- Creating "2차시" after a supplement or asking the server to infer the next
  session could collide with display order and uniqueness constraints.
- The frontend also carried scattered title/order heuristics, so fixing one
  screen could leave another screen using stale sorting or "보강" parsing logic.

## 2. Fix Summary

Backend:

- Added `Session.session_type`, `Session.regular_order`, and
  `Session.display_label`.
- Kept `Session.order` as display order only.
- Added database constraints for:
  - unique display order per lecture/section;
  - unique `regular_order` per lecture/section for regular sessions only;
  - `regular_order` required for regular sessions and forbidden for supplements.
- Added migration backfill that preserves existing regular lesson numbering and
  treats legacy supplement rows as non-regular.
- Reworked session create logic so the server owns all order assignment:
  - auto regular create uses next regular order and next display slot;
  - manual regular create validates the requested nth lesson;
  - supplement create accepts `insert_after_order` and shifts later display
    rows.
- Updated progress, attendance, student app, messaging, media, and result paths
  to consume regular order/display label instead of reinterpreting display
  order.
- Kept the refactor boundary clean by removing the new cross-domain direct
  import from the progress regression test.

Frontend:

- Added `src/shared/product/sessions/sessionOrdering.ts` as the session ordering
  and label SSOT.
- Updated the admin session create modal to support:
  - regular auto numbering;
  - regular manual nth-lesson numbering;
  - supplement insertion after an existing display position.
- Removed scattered session sorting/labeling logic from admin, teacher, student,
  community, exam, result, and video surfaces.
- API request contracts now send semantic fields (`session_type`,
  `regular_order`, `insert_after_order`) and let the backend calculate display
  `order`.

- The model now separates semantic lesson number from display placement.
- The create path is centralized in the backend view layer, and serializers
  reject duplicate or invalid regular/supplement combinations.
- Progress comparisons now use regular order, which preserves lesson hierarchy
  when a supplement is displayed between two regular sessions.

Tenant isolation audit:

- Session creation still resolves through the tenant-scoped lecture and section
  relationship.
- The production smoke used Tenant 1 and cleaned its tagged temporary lecture.
- No new unauthenticated or cross-tenant lookup path was added.

Frontend integration audit:

- The frontend no longer calculates display order for create requests.
- Sorting and labels are centralized in the shared session ordering helper and
  consumed by admin/teacher/student/community/exam/result/video surfaces.

Destructive and edge-case audit:

- Manual regular numbering rejects duplicate nth lessons.
- Supplements cannot receive regular numbers.
- Display order shifts happen transactionally with row locking before insert.
- Legacy rows are backfilled by migration rather than left to title parsing at
  runtime.

UX/accessibility audit:

- Admin create flow supports both simple auto-numbering and explicit nth-lesson
  control.
- Supplement placement uses selectable positions instead of requiring users to
  guess raw order numbers.

Security audit:

- No secret, credential, or provider integration path changed.
- No user-controlled HTML rendering path was introduced.
- The touched API paths continue to rely on the existing authenticated DRF view
  and tenant-scoped queryset patterns.

## 6. Structural Verdict

The session system now has one durable source of truth:

- `regular_order` answers "which nth regular lesson is this?";
- `order` answers "where does this row display?";
- `display_label` answers "what should users see?";
- frontend ordering/labels go through one shared helper.

This removes the previous patch-prone title parsing and duplicated ordering
heuristics while preserving the UX the user asked for: direct nth-lesson
creation, automatic n+1 creation, and supplement lessons that can appear between
regular lessons without becoming part of the regular lesson sequence.
