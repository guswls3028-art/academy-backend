# V1.2.15 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-01 KST
**Scope:** OMR late AI answer auto-recovery in the existing 5-minute recovery job

This release hardens the June 1 tchul OMR incident class beyond the immediate
manual recovery. V1.2.14 made a late AI callback safe to apply. V1.2.15 makes
the existing `academy-v1-recover-stuck-omr` recovery loop find that same state
automatically, so a teacher's manual student match no longer has to be followed
by operator intervention if the AI worker finishes late.

## 1. Automatic Late Answer Recovery

- Added `academy.application.use_cases.omr.late_answer_recovery`.
- The use case scans recent OMR exam submissions that are already
  `answers_ready` or `done`, have a selected `enrollment_id`, have zero
  `SubmissionAnswer` rows, and have a completed AI job result with answers.
- Recovery reuses the canonical mapper and grader path:
  `apply_omr_ai_result()` hydrates the answers and `grade_submission()` regrades
  the submission when it returns to `answers_ready`.
- Recovery is re-entrant: submissions that already gained answers, lost their AI
  result, or still have no answer payload are skipped instead of overwritten.
- Each successful recovery records `meta.late_ai_answer_recovery` with the actor,
  AI job id, source status, and answer count.

## 2. Scheduled Command Integration

- Extended `recover_stuck_omr_submissions` to run late-answer recovery after the
  existing stuck-state auto-fail pass.
- Added command options:
  - `--skip-late-ai-answer-recovery`
  - `--late-lookback-days` (default 14)
  - `--late-limit` (default 100)
- The command output now includes:
  `late_ai_detected`, `late_ai_recovered`, `late_ai_skipped`, and
  `late_ai_failed`.
- The existing EventBridge rule `academy-v1-recover-stuck-omr` remains enabled at
  `rate(5 minutes)`, so no new infrastructure was required.

## 3. Tests

- Added regression coverage for the exact state shape: a manually matched OMR is
  graded to `done` with zero answers, then a late DONE AI result is recovered,
  persisted, and regraded to the expected score.
- Added dry-run coverage proving detection does not write answers or scores.
- Added management-command coverage proving the scheduled command invokes the
  new recovery pass.
