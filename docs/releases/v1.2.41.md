# V1.2.41 Release Notes

**Status:** Production-deployed and E2E-verified
**Release date:** 2026-06-07 KST
**Scope:** Clinic remediation projection fix, account recovery/staff reset real-use proof, and student-domain misuse guardrails

This release closes the remaining student-domain launch gate items that were
still open after v1.2.40: clinic remediation, account recovery/staff password
reset real-use, and beginner/misuse behavior checks.

## 1. Decision

Student-domain technical launch gate is **GO for staged broad expansion**.

No known P0 student-domain blocker remains for the launch decision captured in
`docs/refactor/student-domain-launch-readiness.md`. Run promotion as a staged
ramp with provider balance/template monitoring, worker health checks, E2E
residue probes, and post-deploy account/clinic canary reruns.

## 2. Change Summary

- Fixed clinic target regeneration after remediation: once the same source
  evidence is resolved by `EXAM_PASS`, `HOMEWORK_PASS`, `MANUAL_OVERRIDE`,
  `WAIVED`, or `SOURCE_REMOVED`, the trigger service no longer reopens a new
  unresolved ClinicLink from the original failed result.
- Added regression proof that exam-pass remediation does not recreate the failed
  ClinicLink and that remediated student result projections return
  `clinic_required=false`.
- Added account notification log redaction regression proof for account/password
  reset Alimtalk bodies.
- Added production frontend canaries:
  - `e2e/student/clinic-remediation-realuse.spec.ts`
  - `e2e/auth/account-recovery-realuse.spec.ts`
  - `e2e/student/student-domain-guardrails.spec.ts`
- Updated student-domain launch-readiness and Phase 2 audit SSOT from HOLD to
  staged broad-expansion GO.

## 4. Real-Use Proof

Clinic remediation:

- Admin fixture created lecture/session/student/enrollment/exam/question/answer
  key with production API.
- Student submitted a failing result (`20/100`), creating a ClinicLink target.
- Student browser saw the clinic target, booked a clinic session, and saw the
  booking reflected in `내 일정`.
- Staff API approved/completed the participant and submitted a clinic retake
  score of `90/100`.
- Student result projection reached `remediated=true`, `final_pass=true`,
  `clinic_required=false`; the unresolved clinic target disappeared.
- Student result and grades UI showed the remediated/pass state.

Account recovery and staff reset:

- Public login modal sent a real password-recovery Alimtalk to controlled
  recipient `01031217466`.
- Student account notification log reached `sent` and kept the full phone number
  masked.
- Old password remained valid after public recovery dispatch.
- Staff reset changed only the generated E2E student password, verified JWT login
  with the changed password, restored the original password, and deleted the
  generated student fixture.
- Temporary-password activation is not read from API/logs by design. Backend
  lifecycle tests cover activation and must-change behavior; the canary supports
  optional activation when the controlled device supplies
  `E2E_ACCOUNT_RECOVERY_TEMP_PASSWORD`.

Beginner/misuse guardrails:

- Unauthenticated and fake-token student protected routes fail closed to the auth
  entry.
- Student logout followed by browser back does not restore protected student
  data.
- Student dashboard, grades, exams, clinic, submit, and notifications pages do
  not horizontally overflow at a 390px mobile viewport.
