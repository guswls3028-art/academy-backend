# V1.5.0 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-06-24 KST
**Scope:** Production cost scaling baseline: API 1-to-3 autoscaling, workers 0-to-needed scale-out

This infra release moves the production baseline from always-on spare capacity
to a cost-first dynamic scaling posture. API keeps one healthy instance during
normal traffic and can scale out to three. Messaging, AI, and tools workers can
sit at zero and scale out only when queue depth requires work.

## 2. Change Summary

- API ASG baseline is now `min=1 desired=1 max=3`.
- API ASG has target-tracking autoscaling on `ASGAverageCPUUtilization` at 55%
  with scale-in enabled.
- API deploy workflow temporarily creates refresh headroom, then restores the
  cost baseline after a successful refresh.
- Messaging worker baseline is now `min=0 desired=0 max=3`.
- Tools worker baseline is now `min=0 desired=0 max=2`.
- AI worker remains scale-to-zero at `min=0 desired=0 max=5`.
- Messaging/tools scale-in alarms now use total queue work
  `visible + inflight + delayed`, so workers do not scale in while hidden or
  delayed messages remain.
- Production canary now treats target-tracking and SQS scaling alarms as
  control-plane alarms, not service failures.
- Production Django canary accepts an explicit idle worker mode when the
  messaging ASG is actually `min=0 desired=0`, so stale heartbeat is not a
  failure during intentional scale-to-zero.
- SSOT, deploy scripts, Terraform reference files, cleanup scripts, deploy
  verification, and infra docs were updated to match the new dynamic policy.

## 6. Residual Risk

- Worker jobs can have EC2 cold-start latency when queues move from zero to
  active. This is the intended cost tradeoff.
- API steady state has one instance, so instance-level redundancy is restored
  only after autoscaling or deploy headroom creates extra capacity.
- The API CPU target of 55% is intentionally conservative but may need tuning
  after real traffic observations.
