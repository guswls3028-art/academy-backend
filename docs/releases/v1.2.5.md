# V1.2.5 Release Notes

**Status:** production-deployed; final seal pending optional device-body content validation  
**Release date:** 2026-05-23 KST  
**Scope:** student registration approval orchestration SSOT

## 1. Registration Approval SSOT

- Added `apps.domains.students.services.registration_approval.approve_registration_request`.
- Approval, bulk approval, and auto-approval now share the same durable use case:
  row lock -> pending-state guard -> signup password hash transfer -> student account graph creation -> `pending -> approved`.
- `StudentRegistrationRequest` state transition and `create_student_account(password_hash=...)` now happen in one transaction.
- HTTP response shape remains in `RegistrationRequestViewSet`.
- Approval AlimTalk dispatch remains outside the durable transaction and is nonfatal after commit. A notification transport failure no longer hides an already committed approval as a 500.

## 2. Production QA Finding And Fix

- First production QA on hakwonplus auto-approve exposed a PostgreSQL-specific locking bug:
  `FOR UPDATE cannot be applied to the nullable side of an outer join`.
- Root cause: `select_for_update()` was combined with `select_related("student")`, and `student` is nullable on `StudentRegistrationRequest`.
- Fix: the approval service now locks only the registration request row and avoids joining the nullable student relation in the locking query.
- The failed disposable pending request from the first QA attempt was deleted after confirmation.
