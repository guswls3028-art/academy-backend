## 2. Incident and Root Cause

- The API and infrastructure remained healthy. The incident affected accounts
  whose tenant-local display identifier was shared by more than one active
  membership, including expected student/parent phone-number overlap and legacy
  owner/administrator identifiers.
- A 2026-07-14 authentication hardening change rejected any identifier with
  multiple candidate rows before evaluating the supplied password. Distinct
  passwords could therefore no longer select the intended account.
- A read-only production query found 40 duplicate display-identifier groups and
  80 affected account rows across tenants. No database row was changed while
  diagnosing or closing the incident.

## 3. Fix and Security Contract

- Tenant-scoped login lookup now returns all active membership candidates for an
  exact display identifier. Authentication checks the supplied password against
  those candidates and continues only when exactly one candidate matches.
- The selected account must still have an active tenant membership, active user
  state, valid role profile, and tenant-bound JWT claim.
- A same-identifier, same-password collision remains intentionally fail-closed.
  Pending password-reset state is consumed only for the uniquely matching
  account.
- Login throttling remains in place, bounding the extra password checks for the
  small candidate set without weakening tenant isolation.
