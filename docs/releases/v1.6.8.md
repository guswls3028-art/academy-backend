# V1.6.8 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-07-09 KST
**Scope:** Mandatory student/parent account Alimtalk notices + Ymath block confirmation

This patch release makes account credential notices the system default for
student and parent account creation or credential changes. Staff UI opt-outs
and legacy skip flags are ignored on the backend, profile/account mutations roll
back when a required notice cannot be delivered, and Ymath tenant id `4` remains
fully blocked from Alimtalk/SMS sends until the academy owner approves rollout.

## 1. Decision

GO for production use.

Backend and frontend `main` are deployed. Backend API runtime verification is
GO, frontend production E2E is green, controlled real Alimtalk sends reached the
provider only at `01031217466`, and production Ymath messaging remains disabled
for new sends.

## 2. Change Summary

- Added a shared student-domain account notification helper for required
  student/parent account notices.
- Forced welcome/account notices for direct student creation, bulk JSON
  import, conflict-create flows, staff password resets, student/parent self
  password changes, ID changes, and phone/account changes.
- Removed the staff-facing frontend toggles that previously skipped welcome or
  reset notices.
- Ignored legacy `send_welcome_message=false` and `skip_notify=true` request
  flags for student/parent account notice paths.
- Added fail-closed rollback behavior: if a required account notice fails, the
  profile/account/password mutation returns an error and the attempted account
  change is reverted.
- Covered the blank student-phone case: the system creates separate student and
  parent accounts, not a shared parent-student account. When the student phone
  is registered later, the student account notice is sent with password shown as
  `변경되지 않음` when unchanged.
- Added frontend E2E account-notice safety guard so real Alimtalk E2E sends can
  only target the controlled phone `01031217466`.
- Kept the v1.6.7 Ymath temporary full messaging shutdown in force.

## 5. Audit Lenses

- Account-notice audit: backend is authoritative; frontend skip controls are
  removed, but old clients cannot bypass required student/parent notices.
- Transaction audit: required account-notice failure rolls back the affected
  credential/profile mutation instead of silently leaving an unsent account
  change behind.
- Tenant isolation audit: Ymath tenant id `4` stays blocked by the centralized
  messaging policy from enqueue through worker send.
- Real-send safety audit: production verification used only controlled phone
  `01031217466`; the E2E guard blocks non-controlled real Alimtalk sends.
- Blank-phone audit: tests confirm blank student phone produces separate
  student and parent accounts, then later student phone registration sends the
  student account notice.

  approves Alimtalk use.
- Controlled E2E generated several real notices to `01031217466`; this is
  expected evidence for the production Alimtalk path.
- The production provider verification directly exercised the student account
  trigger, while the exact later-phone-registration HTTP path is covered by
  focused backend tests.
