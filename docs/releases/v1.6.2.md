# V1.6.2 Release Notes

**Status:** Production-deployed / GO
**Release date:** 2026-07-07 KST
**Scope:** Multi-user race-condition hardening + production real-use concurrency verification

This patch release closes a focused multi-user race audit across public
community flows, video folders, attendance writes, messaging auto-send config
updates, and the production API deploy path.

## 1. Decision

GO for production use.

Backend and frontend `main` are deployed. Production canary, deploy
verification, frontend quality gate, and controlled real-use concurrency checks
all passed after cleanup. The release also fixes a deploy-time API headroom race
found during the production test itself.

## 2. Change Summary

- Added race-safe public like toggles for landing board, review, and reply
  surfaces so double-click and concurrent-user actions converge on valid
  counters.
- Hardened public report creation with row locking so duplicate anonymous
  reports race to one create and one conflict instead of duplicate rows.
- Hardened landing-public like counter decrement signals with non-negative
  database expressions.
- Serialized video folder creation by locking the tenant/root or parent folder
  boundary and returning conflict on duplicate concurrent creates.
- Added attendance row locks for update, partial update, delete, and bulk
  present writes.
- Hardened messaging auto-send config writes with row locks and changed the
  frontend to send partial config patches instead of stale full-row payloads.
- Fixed the API deployment workflow race found in production validation:
  target-tracking scale-in could drop deploy headroom from 2 to 1 while a new
  ALB target was still warming, causing a short 502 window. API refresh now
  uses temporary `min=2` headroom plus `MaxHealthyPercentage=200`, then restores
  the cost baseline.
