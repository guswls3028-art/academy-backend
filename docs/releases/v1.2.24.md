# V1.2.24 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-02 KST
**Scope:** OMR tenant 1 real-use canary + ungraded-exam clinic trigger guard

This release closes a hidden production bug found during Tenant 1 OMR real-use
verification. OMR recognition/scoring itself passed, but the progress pipeline
was treating a missing result from another exam in the same session as a scored
failure. That could create premature ClinicLink rows and automatic clinic result
notifications while the remaining exam was still being graded.

## 1. Root Cause

- `SessionProgress.exam_meta.exams` correctly includes every exam in the
  session, including rows where `score=None` and `no_result=True`.
- `ClinicTriggerService.auto_create_per_exam` previously interpreted those
  missing rows as failed exams because `passed=False`.
- In multi-exam sessions, grading the first exam could therefore create an
  automatic ClinicLink before the second exam existed as a Result.
- Once the second exam passed, the system could then resolve that premature
  ClinicLink and trigger a clinic result notification.

## 2. Fix Summary

- `ClinicTriggerService.auto_create_per_exam` now skips exam rows where:
  - `no_result=True`
  - or `score is None`
- Missing/ungraded exams still keep the session incomplete through
  `SessionProgress.completed=False`.
- Missing/ungraded exams are no longer treated as scored failures and no longer
  create ClinicLink/remediation/clinic-result notification side effects.
- Added regression coverage for both sides:
  - missing result does not create an automatic ClinicLink.
  - a real scored failed exam still creates an automatic ClinicLink.

## 4. Operational Note

The first Tenant 1 canary, before this fix was added, found the bug by creating
a premature ClinicLink in a two-exam session. That run also produced two
`clinic_result_notification` success logs to generated canary phone values with
`amount_deducted=0.00`.

Immediate containment:

- checked the messaging queue for those generated numbers and found no pending
  messages to delete.
- replaced the canary student's phone data during investigation.
- cleaned up all canary rows and the two canary notification logs.
- reran the canary after deployment with controlled contact routing and verified
  zero clinic links and zero notification logs.
