# V1.2.18 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-02 KST
**Scope:** OMR objective multi-mark exact grading and manual score hardening

This release fixes the reported multi-answer objective scoring issue: for a
question whose correct answer is `2,3`, marking only `2` or only `3` must be
wrong; only the exact required set should be correct. Alternative answers such
as `2|3` remain supported as "either one is correct".

## 1. Root Cause

- Answer-key registration already supported multi-select objective answers and
  stored required multi-mark answers as comma-separated values such as `2,3`.
- The backend canonical matcher already treated comma-separated values as a
  required set and pipe-separated values as alternatives.
- The admin student result drawer had its own local choice matcher:
  - it split both comma and pipe as candidate choices,
  - it required the student's answer to be a single choice,
  - therefore `2` looked correct for a required `2,3` answer.
- The drawer's manual edit UI also behaved as a single-choice control, so a
  teacher could not enter a multi-mark student answer from that drawer.
- The manual item score API accepted the frontend-provided score, so an old or
  buggy client could still save a partial required answer as full score.

## 2. Fix

- Added a shared frontend objective answer matcher:
  - comma / semicolon / plus / ampersand mean all selected marks are required,
  - pipe / `또는` / `혹은` / `or` mean alternatives,
  - student answers are compared as exact sets, so order does not matter.
- Updated `StudentResultDrawer`:
  - display scoring no longer treats `2,3` as `2` or `3`,
  - edit mode can toggle multiple choice bubbles,
  - score preview uses the exact same required-set semantics.
- Hardened the backend manual item score endpoint:
  - for objective answer-key questions, the server recomputes `score` and
    `is_correct` from `AnswerKey`,
  - `answer="2"` with correct `2,3` is forced to `score=0`,
  - `answer="2,3"` with correct `2,3` is awarded full item score.
- Stabilized the production frontend E2E gate by making session blocks navigate
  directly to the attendance tab instead of relying on a nested route redirect.

## 3. Tests

Backend:

- Added regression coverage for server-side manual item scoring with a required
  multi-mark objective answer.
- Existing answer-matching tests continue to assert:
  - `answer_matches("3", "1,3") == False`
  - `answer_matches("3", "1|3") == True`

Frontend:

- Added a shared matcher used by the score drawer instead of duplicating local
  split logic.
- The score drawer edit flow now stores multiple selected choice bubbles as a
  canonical comma-separated answer.

## 5. Notes

- Required multi-mark answers and alternative answers are intentionally different:
  - `2,3` means both 2 and 3 must be selected.
  - `2|3` means either 2 or 3 is accepted.
- This fix covers both the visible admin score drawer and the server-side manual
  score write path, so the same bug cannot be reintroduced by a stale frontend
  bundle or old client request.
