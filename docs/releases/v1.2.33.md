# V1.2.33 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-05 KST
**Scope:** OMR result projection cleanup + API liveness/readiness hardening

This release closes the OMR cleanup/refactor work with production proof. It
removes obsolete result-grading compatibility code, repairs stale legacy result
projections, and separates API liveness from database readiness so a DB
readiness incident no longer makes the container or ALB lifecycle flap.

## 1. Root Cause

- OMR score shape had already been centralized, but old result-domain files and
  compatibility helpers still suggested alternate grading paths.
- Tenant 2 submissions `301`, `302`, and `303` had correct canonical `Result`
  rows, but stale compatibility projections:
  - `ExamResult` still showed old objective scores;
  - `ExamAttempt.meta.initial_snapshot` still held old totals.
- A separate production incident happened during AWS RDS minor upgrade
  `15.16.R1 -> 15.17.R1`:
  - RDS itself accepted direct connections;
  - RDS Proxy target stayed `UNAVAILABLE / PENDING_PROXY_CAPACITY`;
  - Docker `HEALTHCHECK` still used `/health`, so DB readiness trouble could
    mark the API container unhealthy even though `/healthz` was alive.

## 2. Fix Summary

- Removed dead/duplicate results code:
  - old serializer/view/service files that no production route used;
  - dead manual override/finalize helper methods;
  - tests that only protected removed legacy behavior.
- Documented the current results SSOT:
  - canonical user-visible records are `Result`, `ResultItem`, `ResultFact`,
    and `ExamAttempt`;
  - `ExamResult` is a compatibility snapshot only.
- `sync_result_from_exam_submission()` now also repairs:
  - legacy `ExamResult` DRAFT projection;
  - attempt `initial_snapshot` when it points at the same submission and carries
    stale total/max values.
- Docker API healthcheck now calls `/healthz`; `/health` remains readiness and
  DB smoke.
- Infrastructure docs and SSOT now consistently say:
  - ALB/Docker liveness: `/healthz`;
  - DB readiness: `/health`.
- Runtime stabilization:
  - `/academy/api/env` and `/academy/workers/env` were safely backed up and
    updated to direct RDS `DB_HOST=academy-db...` while RDS Proxy target remains
    unavailable;
  - API `DB_CONN_MAX_AGE=5`;
  - worker `DB_CONN_MAX_AGE_WORKER=0`;
  - RDS engine SSOT updated to actual `15.17`.

## 5. Audit Verdict

Backend domain audit:

- OMR score interpretation now flows through the score-shape SSOT and result
  sync path.
- Legacy `ExamResult` is still present only as compatibility projection, not as
  a separate scoring policy owner.

Tenant isolation audit:

- Tenant 2 production repair was constrained to submissions `301`, `302`, and
  `303`.
- Student matching facts stayed within the same tenant/enrollment IDs.

Infrastructure audit:

- Liveness and readiness are now separated in ALB docs, Docker image behavior,
  and runtime proof.
- RDS Proxy remains unhealthy and is intentionally not the active runtime path
  until recovered. The service currently runs through direct RDS, verified by
  `/health` and direct SQL probes.

- `academy-db-proxy` target is still `UNAVAILABLE / PENDING_PROXY_CAPACITY`.
  This is not blocking current OMR/API operation because API and worker env are
  on direct RDS, but it should be repaired before re-enabling Proxy as runtime
  DB host.
