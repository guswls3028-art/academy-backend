# V1.2.47 Release Notes

**Status:** Production-deployed and tenant 2 OMR layout verified
**Release date:** 2026-06-09 KST
**Scope:** OMR 1/2/3-column layout source of truth and regression coverage

This release removes the main recurrence risk around OMR forms whose objective
answer area expands from one column to two or three columns as the question
count changes.

## 1. Decision

OMR layout generation remains **GO** for up to 60 objective questions on one
sheet.

The source of truth for objective column ranges is now one backend helper used
by generated recognition meta, HTML preview data, and PDF rendering.

## 2. Change Summary

Backend:

- Added `build_mc_column_ranges()` in the OMR meta generator.
- `build_omr_meta()`, `OMRHtmlRenderer`, and `OMRPdfRenderer` now use the same
  column range helper instead of duplicating 20/40/60 threshold logic.
- `OMRDocument.validate()` now calls layout validation against the actually
  rendered essay area, including decorative essay space.
- Deploy-gate smoke now covers 20/21/40/41/60 objective question boundaries.
- OMR matrix tests now assert:
  - 20 -> 1 column,
  - 21/40 -> 2 columns,
  - 41/60 -> 3 columns,
  - HTML/PDF renderers call the shared column range helper,
  - generated meta, rendered PDF, marking overlay, scanner-like distortion,
    A4 alignment, answer detection, and identifier detection succeed at layout
    edges.

## 4. Residual Risk

- The root recurrence risk from duplicated OMR column threshold logic is removed.
- Remaining practical risk is image-quality dependent: very poor scans can still
  fail recognition, but that is scan quality/manual-review behavior, not a
  1/2/3-column layout mismatch.
