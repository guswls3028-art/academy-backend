# V1.8.1 Release Notes

**Status:** Production-deployed / GO

**Release date:** 2026-07-14 KST

**Scope:** Alimtalk-only business-contract seal, tenant-bound billing and
queue integrity, messaging UX simplification, and deployment-propagation
recovery

This patch closes the messaging-focused follow-up to the whole-product V1.8.0
audit. The release treats provider delivery, approved Kakao envelopes, business
tenant ownership, charging, permissions, and user-facing wording as one
contract rather than independent implementation details.

## 1. Decision

GO for production use.

The final backend deployment, runtime digest verification, production canary,
frontend quality/deploy gate, tenant availability checks, and real-use E2E
round trips passed. The release-seal re-audit found no remaining Critical,
High, or Medium finding in the changed messaging, tenant, billing, and frontend
surfaces.

## 2. Alimtalk Business Contract

- Closed SMS/LMS and legacy provider-management paths. Supported user-facing
  sends are Alimtalk only, and the worker also fails legacy SMS payloads closed.
- Unified automatic and manual delivery through approved Kakao envelopes.
  Signup flows retain their dedicated approved templates; mapped operational
  categories use the approved common envelopes; unmapped categories do not
  silently fall back to an unrelated template.
- Removed self-service PFID, sender, provider-key, template submission, review,
  synchronization, and charging controls that contradicted the centrally
  operated owner channel.
- Restricted manual sending and diagnostics to the intended roles, added
  recipient/request/context/preview bounds, and redacted sensitive diagnostics.
- Made automatic-send readiness depend on the envelope that will actually be
  delivered, not merely on the existence of an editable phrase.

## 3. Tenant and Billing Integrity

- Separated the physical owner channel from the source business tenant. The
  owner remains the provider/log envelope owner, while cost, balance debit,
  refund, and business attribution remain with the source tenant.
- Added a dedicated HMAC tenant-binding key and signed canonical payloads.
  Workers reject missing or contradictory tenant bindings before provider calls
  or charging.
- Synchronized the dedicated key across API and worker SSM environments,
  drained the queue, and enabled `MESSAGING_TENANT_BINDING_ENFORCED=true` in the
  production messaging worker.
- Preserved Ymath (tenant 4) as an intentional operational messaging hold. The
  production policy returns `messaging_disabled=true`, the UI presents an
  operational stop instead of a configuration error, and all enqueue/send
  paths remain blocked.

## 4. Messaging UI/UX

- Reduced the message experience to four user concepts: saved phrases,
  automatic-send switches, delivery history, and operational status.
- Clarified the distinction between editable content and Kakao-approved
  envelopes. Exam, score, assignment, and retake flows now show the actual
  trigger context with `카카오 승인 공용 양식` rather than leaking an unrelated
  internal provider-template label.
- Fixed switch rollback, readiness/error feedback, scheduled-limit display,
  mobile density, and keyboard/accessibility behavior.
- Kept delivery-channel selectors visibly fixed to `알림톡`, matching backend
  enforcement instead of offering non-functional choices.

## 5. Deployment and Browser Recovery

- Migration jobs now atomically refresh `/opt/api.env` from SSM immediately
  before executing the new digest-pinned image. This prevents stale instance
  env files from failing newly introduced production assertions.
- The frontend deployment gate waits for the exact deployed version and the
  route chunks used by production E2E to return successfully for three
  consecutive checks.
- The frontend error boundary performs bounded recovery for transient chunk
  propagation mismatches and never enters an unbounded reload loop.

## 8. Intentional State

- Ymath messaging remains OFF until the product owner explicitly removes the
  operational hold. This is not treated as missing configuration or silently
  bypassed through the owner channel.
- Kakao approval still constrains the physical envelope. Users may edit the
  allowed content/context, but the product does not imply that arbitrary saved
  phrases are independently approved provider templates.
- No release can prove the permanent absence of all future defects. This seal
  records that the current Critical/High/Medium re-audit findings were closed
  and that the production evidence above passed at release time.
