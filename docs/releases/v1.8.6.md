# V1.8.6 Release Notes

**Status:** Production-deployed / GO

**Release date:** 2026-07-20 KST

**Scope:** Multi-subject score-report intake, cumulative score operations,
evidence lifecycle integrity, and remaining-risk closure

## 1. Decision

GO for production use.

The results console and student score-report workflow now cover academy tests,
school semester exams, nationwide mock exams, KICE mock assessments, performance
assessments, and other named exams without mixing incompatible score sources.
One original report can submit up to 20 subjects atomically. Staff review the
report as one bounded group, while only verified subject rows enter cumulative
analytics.

## 2. Business and UX Contract

- School exams retain academic year, semester, and first/second written-exam
  rounds. Nationwide and KICE mocks retain their own month-based source types;
  performance and other exams require a custom name and exam date.
- A student uploads one image/PDF and enters all subjects shown on that report.
  Duplicate subjects or one invalid row reject the whole request before any
  score row is written.
- The admin console supports tenant-wide name/school/lecture search, grade,
  lecture, source, subject, score band, trend, period, and sort controls. Search
  is debounced; student and review queues are server-paginated.
- Review is evidence-grouped. Staff can verify or reject remaining pending rows,
  explicitly confirm a 5-grade or 9-grade school scale, and void previously
  verified history with a required reason.
- Desktop and mobile admin/student screens were rendered and visually reviewed.
  Loading, empty, retry, partial-review, and narrow-screen chart states are
  explicit.

## 3. Data Integrity and Recovery

- Migration `results.0014_studentreportedscore_exam_name_and_more` adds flexible
  exam names, the voided terminal state, plural evidence linkage, and a database
  constraint requiring evidence for pending or verified scores.
- Evidence cannot be deleted, overwritten, moved directly, or moved through a
  parent folder while any score audit row references it. Storage adapter absence
  or provider failure is fail-closed and preserves the database row.
- Student, score, tenant, and evidence boundaries are revalidated after row locks.
  A stale concurrent review returns conflict instead of overwriting a newer
  transition.
- Exact UUID-tagged production canaries can recover ambiguous active submissions,
  detach evidence only after terminalization, and remove both provider objects and
  audit rows. Cleanup is marker-bounded and residue-checked.
- Flexible exams use their exam date in replacement identity, so two same-named
  assessments on different dates remain separate cumulative points.

## 4. Performance and Residual Risk

- Results responses use a tenant/filter/page/data-version cache with a 300-second
  reuse window; score changes alter the version key immediately. Search debounce,
  roster pagination, and bounded review groups prevent unbounded UI payloads.
- No Critical or High finding remains after independent business, destructive
  lifecycle, contract/release, and frontend UX audits.
- The only accepted Medium risk is a cold, previously unseen filter combination:
  its first request still computes tenant-wide aggregate versions and summaries.
  Warm requests are cached. Production p95/query-plan/load telemetry should be
  reviewed as score volume grows; move to write-side generations or pre-aggregates
  only if measured thresholds fail.
