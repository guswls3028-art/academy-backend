# V1.2.27 Release Notes

**Status:** production-deployed
**Release date:** 2026-06-02 KST
**Scope:** Student video progress access hardening

This release hardens the student video watch-history path after the v1.2.26
progress enrollment fix. It closes adjacent failure classes around parent
read-only progress echoes and explicit video access blocks.

## 1. Root Cause

- v1.2.26 fixed progress persistence to the correct video lecture enrollment.
- Additional hardening found that the parent progress endpoint returned a
  read-only echo before validating the selected child/video enrollment.
- Progress saves used enrollment resolution but did not reject
  `VideoAccess.access_mode=BLOCKED` when the legacy `rule` field remained
  `free`.
- Student playback still had a legacy-rule check before the SSOT access mode
  result, so a BLOCKED access mode could reach later playback URL logic.

## 2. Fix Summary

- Parent progress echo now runs only after the request resolves an allowed
  child enrollment for the target video's lecture.
- Public-video progress echo now checks same-tenant student access before
  returning a non-persistent 200 response.
- Progress POST rejects `AccessMode.BLOCKED` and creates no `VideoProgress`
  row.
- Playback GET rejects `AccessMode.BLOCKED` even when the legacy
  `VideoAccess.rule` value is `free`.
- Regression tests cover:
  - automatic target enrollment progress save,
  - explicit wrong enrollment rejection,
  - BLOCKED progress rejection,
  - BLOCKED playback rejection,
  - unlinked parent rejection,
  - linked parent read-only echo without DB writes.

## 4. Operational Note

The original teacher report was about video watch history, not course
registration. New student video viewing activity now persists only when the
student is allowed to watch the video, and parent accounts can no longer get a
successful progress echo without a valid child/video enrollment context.
