---

## 1. 개요

본 보고서는 코드베이스(`backend/scripts/v1/`, `backend/docs/00-SSOT/v1/`, `backend/apps/support/video/`, `backend/apps/infrastructure/storage/`)를 기준으로 **동영상 서버 인프라가 어떻게 구축되었는지** 실제 설정·스크립트·애플리케이션 로직을 확인한 결과를 정리한 것이다.

동영상 인프라는 **AWS Batch(EC2 기반)** 로 영상 인코딩(FFmpeg HLS)을 수행하고, **Cloudflare R2** 에 원본(raw) 및 HLS 결과물을 저장하며, **DynamoDB** 로 1 video 1 job 락과 multipart 업로드 체크포인트를 관리한다. ECS/EKS는 사용하지 않고, 아키텍처 룰에 따라 **EC2 ASG + Batch 전용** 구성이다.

## 3. 동영상 인프라가 구축되는 방식

### 3.1 배포 플로우 (deploy.ps1)

정식 배포 시 다음 순서로 동영상 관련 리소스가 Ensure 된다.

1. **Ensure-BatchIAM** (`resources/iam.ps1`)  
   - Batch 서비스 역할, ECS 인스턴스/실행 역할, **Job 역할**(`academy-video-batch-job-role`) 생성·갱신  
   - Job 역할에 `policy_video_job_role.json` inline 적용 (SSM, ECR, CloudWatch Logs/Metric, Batch SubmitJob/TerminateJob/DescribeJobs, **DynamoDB job-lock·upload-checkpoints**)

2. **Ensure-DynamoLockTable** / **Ensure-DynamoUploadCheckpointTable** (`resources/dynamodb.ps1`)  
   - `academy-v1-video-job-lock`: PK=`videoId`(S), TTL 속성 `ttl`, PAY_PER_REQUEST  
   - `academy-v1-video-upload-checkpoints`: PK=`uploadId`(S), SK=`partNumber`(N), multipart 업로드 재개용

3. **Ensure-VideoCE** / **Ensure-VideoLongCE** / **Ensure-OpsCE** (`resources/batch.ps1`)  
   - Standard CE: `academy-v1-video-batch-ce` (BEST_FIT_PROGRESSIVE, Spot 혼합, maxvCpus 40, c6g.xlarge)  
   - Long CE: `academy-v1-video-batch-long-ce` (BEST_FIT, On-Demand, maxvCpus 80, c6g.xlarge)  
   - Ops CE: `academy-v1-video-ops-ce` (m6g.medium, maxvCpus 2)  
   - 템플릿: `templates/batch/video_compute_env.json`, `video_compute_env_long.json`, `ops_compute_env.json`

4. **Ensure-VideoQueue** / **Ensure-VideoLongQueue** / **Ensure-OpsQueue**  
   - 각 CE에 1:1 연결된 Job Queue 생성·ENABLED 유지

5. **Ensure-VideoJobDef** / **Ensure-VideoLongJobDef** + Ops JobDef (`resources/jobdef.ps1`)  
   - Worker: `academy-v1-video-batch-jobdef` (2 vCPU, 4096 MiB, timeout 6h), `academy-v1-video-batch-long-jobdef` (12h timeout)  
   - 이미지: ECR `academy-video-worker:latest`, 진입점 `python -m apps.worker.video_worker.batch_main Ref::job_id`

6. **Ensure-EventBridgeRules** (`resources/eventbridge.ps1`)  
   - `academy-v1-reconcile-video-jobs`: rate(1 hour) → Ops Queue → reconcile JobDef  
   - `academy-v1-video-scan-stuck-rate`: rate(1 hour) → Ops Queue → scanstuck JobDef

7. **Ensure-VideoBatchLogRetention**  
   - CloudWatch Logs 보존 기간 설정

**params.yaml 파서 이슈:** `videoBatch.long` 중첩 구조를 deploy의 SSOT 로더가 지원하지 않아, Long CE/Queue/JobDef는 **deploy.ps1만으로는 생성되지 않을 수 있다.** 이 경우 **`ensure-video-long.ps1`** 를 별도 실행해 Long 리소스를 수동 생성한다. (스크립트 내부에서 Long 이름·스펙을 하드코딩하여 Ensure-VideoLongCE/Queue/JobDef 호출.)

### 3.2 SSOT (params.yaml) 기준 동영상 스펙

- **Standard:** maxvCpus 40, c6g.xlarge, rootVolume 200GB, jobTimeout 21600s(6h), Spot 혼합, BEST_FIT_PROGRESSIVE  
- **Long:** maxvCpus 80, c6g.xlarge, rootVolume 300GB, jobTimeout 43200s(12h), On-Demand, BEST_FIT  
- **Ops:** m6g.medium, maxvCpus 2, reconcile/scanstuck 스케줄 rate(1 hour)  
- **DynamoDB:** lockTableName `academy-v1-video-job-lock`, lockTtlSeconds 43200, uploadCheckpointTableName `academy-v1-video-upload-checkpoints`  
- **R2:** videoBatch 하위 r2UploadPartSizeMb 64, r2UploadMaxConcurrency 8, r2UploadMaxAttempts 8, r2UploadCheckpointTable 위와 동일

### 3.3 애플리케이션 측 구축 연동

- **업로드 완료 → Batch 제출**  
  - API: `upload_complete` 뷰에서 `create_job_and_submit_batch` 호출  
  - `batch_submit.submit_batch_job(video_job_id, duration_seconds)`  
  - `duration_seconds >= VIDEO_LONG_DURATION_THRESHOLD_SECONDS`(기본 10800 = 3시간)이면 **Long 큐/JobDef**, 미만이면 **Standard 큐/JobDef** 사용  
  - 설정: `VIDEO_BATCH_JOB_QUEUE`, `VIDEO_BATCH_JOB_DEFINITION`, `VIDEO_BATCH_JOB_QUEUE_LONG`, `VIDEO_BATCH_JOB_DEFINITION_LONG` (SSM `/academy/api/env` 등에서 주입)

- **API EC2 역할**  
  - `academy-ec2-role`에 `policy_api_video_upload.json` inline 적용  
  - Batch SubmitJob/TerminateJob/DescribeJobs/DescribeJobQueues/DescribeJobDefinitions + **DynamoDB video-job-lock** (lock_acquire용 PutItem/ConditionCheckItem 등)  
  - IAM PassRole(batch.amazonaws.com)

- **동영상 스토리지**  
  - R2 버킷 `R2_VIDEO_BUCKET`(기본 academy-video): 원본(raw) 및 HLS 출력  
  - `r2.py`: `delete_object_r2_video`, `delete_prefix_r2_video`로 삭제 처리  
  - Batch 워커는 SSM에서 R2 설정을 읽어 ECR 이미지 내에서 R2 GET(raw) → FFmpeg → R2 PUT(HLS) 수행

## 4. IAM 역할 정리 (동영상 관련)

| 역할 | 용도 | 주요 권한 (코드 기준) |
|------|------|------------------------|
| `academy-video-batch-job-role` | Batch 컨테이너(video worker) | SSM GetParameter, ECR pull, CloudWatch Logs/PutMetric, Batch DescribeJobs/TerminateJob/SubmitJob, **DynamoDB** job-lock·upload-checkpoints (PutItem/DeleteItem/GetItem/UpdateItem/ConditionCheckItem) |
| `academy-ec2-role` (API) | API EC2 인스턴스 | Batch SubmitJob/TerminateJob/DescribeJobs 등, **DynamoDB** video-job-lock (upload_complete 시 lock_acquire) |
| `academy-eventbridge-batch-video-role` | EventBridge → Batch | Ops Queue로 reconcile/scanstuck Job 제출 |

과거 인시던트(incident-video-batch-iam-dynamodb-20260310.md)에서 **Job 역할에 DynamoDB 권한이 없어** 완료 후 lock_release 실패 → stale lock → upload_complete 503 이 발생한 바 있으며, `policy_video_job_role.json`에 DynamoDB 권한을 추가하고 `Ensure-BatchIAM`이 해당 파일을 기준으로 적용하도록 정리되어 있다.

## 6. 데이터 흐름 요약

1. **업로드:** 프론트 → presigned PUT → R2 (raw)  
2. **업로드 완료:** API → DynamoDB lock_acquire(videoId) → VideoTranscodeJob 생성 → Batch SubmitJob(Standard 또는 Long) → 워커 실행  
3. **인코딩:** Batch 워커 → SSM에서 R2/설정 로드 → R2 GET(raw) → FFmpeg HLS → R2 PUT(HLS) → 완료 시 DynamoDB lock_release  
4. **재생:** CDN/r2.dev → R2 GET(HLS)  
5. **삭제:** API → R2 delete_object/delete_prefix (및 DB 정합성 처리)  
6. **복구/관측:** EventBridge 1시간 주기 → Ops Queue → reconcile(DB↔Batch 정합성), scanstuck(Stuck worker 감지)

**결론:** 동영상 서버 인프라는 **단일 SSOT(params.yaml)** 와 **PowerShell Ensure 스크립트(deploy.ps1 → batch.ps1, jobdef.ps1, dynamodb.ps1, eventbridge.ps1, iam.ps1)** 로 구축되며, **AWS Batch(Standard/Long/Ops)** 와 **DynamoDB(락·체크포인트)**, **EventBridge(스케줄)** 가 연동되고, 스토리지는 **R2(academy-video)** 로 구성되어 있다. API는 EC2 역할로 Batch SubmitJob과 DynamoDB lock_acquire를 수행하고, Batch Job 역할은 인코딩 후 DynamoDB lock_release 및 R2 업로드를 수행한다.
