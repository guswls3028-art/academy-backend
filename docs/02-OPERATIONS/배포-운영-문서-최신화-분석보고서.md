## 2. 현재 문서와 실제 코드의 차이

| 항목 | 문서/규칙 내용 | 실제 코드 | 차이 |
|------|----------------|-----------|------|
| **push 후 서버 반영** | CI-CD-분석 “push만으로 반영 안 됨” | 워크플로우에 **deploy-api-refresh** job 있음. push → 빌드 → ECR 푸시 → **instance refresh** 까지 자동 실행. | 분석 문서의 “끊기는 지점”은 deploy-api-refresh 추가 **이전** 기준. 현재는 push만으로도 instance refresh로 반영됨. |
| **배포 방식 명칭** | 배포.md “레거시(원격 자동 배포)” | api-auto-deploy-remote.ps1 주석·Rapid-Deploy-사용법에서는 **Rapid Deploy**로 통일. | 문서에서 **Rapid Deploy**로 통일하고, “레거시”는 “개발 중 빠른 반영용”으로 목적 명시 필요. |
| **배포 2트랙** | 07_deployment_orchestrator, .cursorrules | Formal(deploy.ps1 + CI instance refresh) / Rapid(api-auto-deploy On → cron → 컨테이너만) 두 경로 존재. | 규칙에 **Formal Deploy / Rapid Deploy** 구분, 사용 시점, “언제 무엇을 쓸지” 명시 필요. |
| **에이전트 배포 반영** | .cursorrules “백엔드: 푸시 후 deploy.ps1 실행” | 개발 중에는 Rapid Deploy ON + push만으로도 CI instance refresh 또는 서버 cron으로 반영 가능. | “정식 반영은 deploy.ps1 / 개발 중 빠른 반영은 Rapid Deploy”로 구분해 규칙 반영 필요. |

## 3. Formal Deploy 문서화 필요 항목

- **목적:** 안정 반영, 인프라/Launch Template/userdata/SSM 변경 반영, 릴리즈·하루 마감용.
- **실행 방식:** `pwsh scripts/v1/deploy.ps1 -AwsProfile default`. 순서: Bootstrap(선택), Ensure-Network, Ensure-ECR, Ensure-API-LaunchTemplate, Ensure-API-ASG, Ensure-API-Instance → LT drift 시 `start-instance-refresh` (MinHealthyPercentage, InstanceWarmup). 새 인스턴스 기동 시 UserData로 ECR pull, SSM→/opt/api.env, docker run.
- **특징:** 느리지만 정석. 반영 범위 넓음(API LT, ASG, ALB, Batch, RDS/Redis 확인 등). 새 인스턴스 기동·검증 성격.
- **언제 쓸지:** 인프라 변경 반영 시, 출시 전/후, 안정 반영 필요 시.
- **언제 Rapid로 대체하면 안 되는지:** Launch Template/UserData/SSM/ASG/ALB 변경이 있을 때. env/테넌트 관련 SSM 수정 후 인스턴스에 반영할 때.
- **실행 후 검증:** deploy.ps1 내 After-Deploy Verification(ASG desired/inService, ALB target health, Batch CE/Queue). 필요 시 `run-qna-e2e-verify.ps1`.
- **CI와 관계:** main push 시 CI가 ECR 푸시 후 **deploy-api-refresh**로 API ASG instance refresh까지 수행. 즉 “push만” 해도 Formal 경로의 instance refresh는 CI가 실행함. deploy.ps1은 “전체 인프라 Ensure + API refresh”용.

---

## 4. Rapid Deploy 문서화 필요 항목

- **목적:** 개발 중 빠른 코드 반영, 잦은 수정 대응, API 컨테이너만 빠르게 pull/restart, 2분 감지 기반 자동 반영.
- **실행 방식:**  
  - **ON:** `pwsh scripts/v1/api-auto-deploy-remote.ps1 -Action On -AwsProfile default` → 서버에 cron 등록 (`*/2 * * * *` … `deploy_api_on_server.sh`).  
  - **OFF:** `-Action Off` → cron 제거.  
  - **감지:** `git fetch origin main`, `HEAD` vs `origin/main` 비교. 다르면 `git reset --hard origin/main` 후 `scripts/deploy_api_on_server.sh` 실행.  
  - **서버 동작:** SSM→/opt/api.env, ECR pull academy-api:latest, docker stop/rm academy-api, docker run --env-file /opt/api.env.  
  - **health check:** deploy_api_on_server.sh 내 `curl -sf $HEALTH_URL` (기본 http://localhost:8000/healthz).  
  - **최신 반영 기준:** main 커밋이 갱신된 뒤 cron이 감지(최대 2분). 이미지는 ECR latest(CI가 push한 뒤여야 최신).
- **특징:** 빠름. 인프라 재구성 없음. 개발 생산성용. Formal Deploy 대체 아님.
- **언제 쓸지:** 개발 중 코드 자주 수정할 때, API 컨테이너만 반영하면 될 때.
- **언제 쓰면 안 되는지:** Launch Template/SSM/ASG 등 인프라 변경 시. env/테넌트 SSM 수정만 하고 인스턴스 재기동 없이 쓰면 안 됨(이미 deploy_api_on_server.sh가 SSM→/opt/api.env 하므로 1회 Deploy는 가능).
- **작업 종료 후 OFF:** 자동 push·Rapid Deploy ON 상태 방치 금지. 의도치 않은 main 변경까지 서버에 반영될 수 있음.

## 5. 규칙 반영 필요 항목

- **07_deployment_orchestrator.mdc:** Formal Deploy(진입점 deploy.ps1, CI의 deploy-api-refresh)와 Rapid Deploy(api-auto-deploy-remote.ps1 On/Off, 컨테이너만 반영) 구분. 언제 무엇을 쓰는지 한 줄씩.
- **.cursorrules §4:** 백엔드 수정 시 “정식 반영은 deploy.ps1 또는 CI push(instance refresh)” / “개발 중 빠른 반영은 Rapid Deploy ON 후 push. 작업 종료 시 Rapid Deploy OFF” 명시.
- **09_multitenant_isolation.mdc:** Rapid Deploy 중에도 tenant isolation 완화 금지. 배포 경로(Formal/Rapid)와 무관하게 env는 SSM→/opt/api.env만 사용, 검증은 run-qna-e2e-verify 등 tenant 기준 유지.
- **공통:** tenant resolver, auth, middleware, worker, cache, storage, aggregate, deployment 관련 수정 시 배포 후 검증 필수. 작업 종료 시 auto push / Rapid Deploy ON 방치 금지. 문서와 스크립트 불일치 금지.
