## 2. 현재 멀티테넌트 구조 요약

### tenant resolve 방식
- **1단계:** Host가 `TENANT_HEADER_CODE_ALLOWED_HOSTS`(기본 `api.hakwonplus.com`)이고 `X-Tenant-Code`가 있으면 → `tenant_get_by_code(raw)`로 테넌트 결정. (`resolver.py` 76–94, 111–114)
- **2단계:** `request.get_host()` 정규화값으로 `TenantDomain` 조회 → 해당 tenant 반환. (`_resolve_tenant_from_host`)
- **3단계:** Host에 해당하는 TenantDomain이 없을 때(ALB 내부 Host 등) `X-Tenant-Code`가 있으면 → 코드로 테넌트 결정. (104–131)
- **bypass path:** `/health`, `/healthz`, `/readyz` 등은 tenant=None 허용. (`middleware/tenant.py` BYPASS_PATHS)

### request → tenant binding 흐름
- `TenantMiddleware`가 `resolve_tenant_from_request(request)` 호출 후 `request.tenant` 설정, `set_current_tenant(tenant)` 호출. (`apps/core/middleware/tenant.py` 46–76)
- 예외 시 `TenantResolutionError` → 403/404/500 JsonResponse. 응답에 `X-Tenant-Code`, `X-Tenant-Id` 부여.

### 학생앱/선생앱/API/DB 연결 구조
- **학생앱:** JWT 발급 시 `X-Tenant-Code` 또는 body `tenant_code` 필수. `user_get_by_tenant_username(tenant, username)`으로 해당 테넌트 User만 로그인. (`auth_jwt.py`)
- **선생/스태프:** 동일 API, `request.tenant` 기준. `staffs/views.py` 등에서 `request.tenant`로 repo 호출.
- **DB:** 단일 RDS. tenant 격리는 **애플리케이션 레벨** — 모델에 `tenant` FK, selector/view에서 `tenant=` 필터 사용.

### tenant 기준 데이터 격리 방식
- **Selector/Service:** 대부분 `tenant` 인자를 받아 `Model.objects.filter(tenant=tenant, ...)` 사용. (예: `post_selector.py` `get_post_by_id(tenant, post_id)`, `get_all_posts_for_tenant(tenant)`)
- **View:** `tenant = getattr(request, "tenant", None)` 후 없으면 403 또는 학생 tenant 폴백(community). 생성 시 `serializer.save(tenant=request.tenant)` 또는 동일 의미 유지.
- **TenantQuerySet:** `apps/core/db/tenant_queryset.py` — `for_current_tenant()`, `require_tenant()` 제공. 사용처는 제한적.

### 공용 데이터와 tenant 종속 데이터 구분
- **Tenant, TenantDomain, User(tenant_id), TenantMembership:** 테넌트 또는 멀티테넌트 식별용.
- **도메인 모델(PostEntity, Student, Lecture, Session, Video 등):** tenant FK 또는 session→lecture→tenant 경로로 tenant 종속. 조회 시 tenant 필터 필요.

## 4. 확인된 문제 지점

### 문제 1. 영상 진행률 API에서 Video 단건 조회 시 tenant 미필터 (cross-tenant 정보 노출 가능)

- **현상:** `GET /api/v1/.../progress/` 등에서 Redis 미적중 시 `Video.objects.filter(pk=video_id)`로 Video만 조회한 뒤, 해당 video의 tenant로 Redis를 다시 조회하여 응답에 사용.
- **근거 파일:** `apps/support/video/views/progress_views.py` 91–106행.
- **근거 코드:**  
  `tenant = getattr(request, "tenant", None)` 후 `get_video_status_from_redis(tenant.id, video_id)` 호출.  
  Redis가 None이면 `video = Video.objects.filter(pk=video_id).select_related(...).first()` 로 **tenant 없이** 조회.  
  `if alt_tenant and alt_tenant.id != tenant.id` 일 때 `cached_status = get_video_status_from_redis(alt_tenant.id, video_id)` 로 다른 테넌트 Redis까지 사용하고, 그 결과를 응답에 사용할 수 있음.
- **위험:** 다른 테넌트의 video_id를 아는 경우, 해당 테넌트의 영상 진행률이 노출될 수 있음.
- **심각도:** 중간 (진행률 정보만, 재생 URL 등은 아님).
- **수정 가능 여부:** 즉시 가능. Video 조회 전에 `request.tenant`로 접근 가능한 video인지 검증하거나, Video 조회 시 `session__lecture__tenant=request.tenant` 등 tenant 경로로 필터 추가 후, 일치할 때만 Redis fallback 사용.

### 문제 2. run-api-management-remote.ps1의 env 파일 경로 불일치

- **현상:** 원격에서 management command 실행 시 `--env-file /home/ec2-user/.env` 사용. 정석 배포(Launch Template UserData)는 SSM → `/opt/api.env`만 생성함.
- **근거 파일:** `scripts/v1/run-api-management-remote.ps1` 14행, `scripts/v1/resources/api.ps1` 44–51행.
- **근거 코드:**  
  api.ps1: `ENV_JSON=$(aws ssm get-parameter ...)` 후 `echo ... > /opt/api.env`, `API_ENV_FILE="--env-file /opt/api.env"`.  
  run-api-management-remote.ps1: `docker run ... --env-file /home/ec2-user/.env academy-api:latest ...`
- **위험:** 새 인스턴스 또는 `/home/ec2-user/.env`가 없는 서버에서는 management command 실패. 배포/운영 일관성 저하.
- **심각도:** 중간 (기존에 .env를 수동으로 둔 환경에서는 동작 가능).
- **수정 가능 여부:** 즉시 가능. `/opt/api.env`를 우선 사용하도록 변경하고, 없을 때만 `/home/ec2-user/.env` 사용하거나, 문서에 “정석 배포는 /opt/api.env만 있음” 명시.

---

### 문제 3. X-Tenant-Code fallback의 허용 Host 제한 없음 (규칙 3)

- **현상:** Resolver 규칙 3에서는 Host가 TenantDomain에 없을 때 **아무 Host에서나** `X-Tenant-Code`만 있으면 테넌트를 결정함.
- **근거 파일:** `apps/core/tenant/resolver.py` 125–131행.
- **근거 코드:**  
  `raw = (request.META.get("HTTP_X_TENANT_CODE") or "").strip()`  
  `if raw: tenant = core_repo.tenant_get_by_code(raw)`  
  규칙 1에서는 `TENANT_HEADER_CODE_ALLOWED_HOSTS`로 Host 제한이 있으나, 규칙 3에는 Host 체크 없음.
- **위험:** 내부망/스푸핑 가능한 환경에서 악의적 클라이언트가 임의 Host로 요청하면서 `X-Tenant-Code`만 넣어 다른 테넌트로 해석되게 할 수 있음. 현재는 ALB/Cloudflare 뒤에서만 오므로 실질적 위험은 제한적이나, 원칙적으로 “허용 Host 또는 ALB 내부 Host”만 fallback 허용하는 편이 안전함.
- **심각도:** 낮음 (현재 운영 경로상 제한적).
- **수정 가능 여부:** 즉시 가능. 규칙 3에서도 `request.get_host()`가 허용 목록 또는 ALB/내부용 패턴에 한해 X-Tenant-Code fallback 허용하도록 제한.

---

### 문제 4. is_reservation_cancelled에서 tenant 없이 Reservation 조회

- **현상:** `model.objects.filter(pk=reservation_id).first()` 로 모든 앱의 `Reservation` 모델에 대해 pk만으로 조회.
- **근거 파일:** `apps/support/messaging/services.py` 182–196행.
- **근거 코드:**  
  `for model in apps.get_models():`  
  `if model.__name__ == "Reservation" and hasattr(model, "status"):`  
  `r = model.objects.filter(pk=reservation_id).first()`
- **위험:** Reservation이 tenant FK를 갖는 모델이면, 다른 테넌트 reservation_id로 조회 가능. 호출 경로에서 tenant가 선행 제한되는지 코드만으로는 미확인.
- **심각도:** 낮음 (호출부가 tenant 범위 내 id만 넘기면 무해).
- **수정 가능 여부:** 부분 가능. 호출부가 tenant를 넘기면 `filter(tenant=tenant, pk=reservation_id)` 형태로 수정 권장. Reservation 모델 정의 및 호출처 추가 확인 필요.

## 5. 배포 안정성 점검 결과

### 정석 배포 스크립트 검토 결과
- **scripts/v1/deploy.ps1:** ECR 이미지 빌드는 하지 않고(`-SkipBuild` 기본), GitHub Actions 등 CI에서 푸시한 이미지를 pull하여 배포. Launch Template UserData에 SSM `/academy/api/env` → `/opt/api.env`, `docker pull` 후 `docker run --env-file /opt/api.env` 사용. (`api.ps1` 44–58)
- **일관성:** API ASG는 Launch Template 갱신 시 instance-refresh로 새 인스턴스에 최신 UserData(이미지 URI, SSM 파라미터명) 반영. 새 인스턴스는 `/opt/api.env`만 갖고 기동함.

### 원격 자동 배포 스크립트 검토 결과
- **docs/02-OPERATIONS/배포.md** 10절: `deploy_api_on_server.sh`, `auto_deploy_cron_on.sh` 등으로 서버에서 git pull + 로컬 빌드 + docker run 방식을 설명. 이 방식은 **정석(deploy.ps1 + ECR)** 과 다름. `.cursor/rules/07_deployment_orchestrator.mdc` 에서 “빌드 서버 0대, 이미지는 GitHub Actions OIDC만”이라고 했으므로, 배포.md의 “원격 자동 배포”는 레거시/수동 배포용일 가능성 있음. **문서와 현재 정석 배포 흐름이 혼재**되어 있음.

### env/SSM/ECR/instance refresh 반영 흐름
- **SSM:** `/academy/api/env` JSON → UserData에서 파싱해 `/opt/api.env` 생성. 인스턴스 부팅 시 1회.
- **ECR:** UserData에 치환된 `$ApiImageUri`(ECR full URI:latest)로 pull 후 동일 URI로 `docker run`. instance-refresh 시 새 LT 버전으로 새 인스턴스가 기동하면 최신 이미지 pull.
- **기존 인스턴스:** refresh 전 인스턴스는 재기동하지 않으면 이미지/ env 갱신 없음. MinHealthyPercentage=100이면 새 인스턴스가 healthy 된 뒤 기존 종료이므로, 배포 반영은 “새 인스턴스” 기준으로만 보장됨.

### 배포 후 검증 흐름 적절성
- **run-qna-e2e-verify.ps1:** ALB DNS 조회 후 `http://ALB_DNS` + `/opt/api.env`로 `manage.py verify_qna_e2e` 실행. 실제 서버 env·이미지·경로와 일치함. 적절함.
- **run-api-management-remote.ps1:** `/home/ec2-user/.env` 사용으로 **실제 정석 배포 서버 구조와 불일치**. 위 문제 2와 동일.

### 문제점 및 보강 필요점
- run-api-management-remote의 env 경로를 `/opt/api.env`로 통일하거나, “정석 배포 서버에는 /opt/api.env만 있음”을 주석/문서에 명시.
- 배포.md 10절과 07_deployment_orchestrator.mdc 관계 정리: 어떤 경로가 정석인지, 원격 자동 배포 스크립트가 현재 사용 중인지 명시.
- 배포 후 “실제 실행 이미지 digest” 수집은 `Invoke-CollectRuntimeImagesReport`로 수행 중. CI digest와 불일치 시 보고서에 명시되는 구조는 적절함.

## 6. 우선순위별 보강안

### A. 즉시 수정해야 할 항목
1. **progress_views.py:** Video 조회 시 tenant 범위 제한. `request.tenant`로 접근 가능한 video만 조회하거나, `Video.objects.filter(... session__lecture__tenant=request.tenant)` 등으로 필터 후 fallback 로직 수행.
2. **run-api-management-remote.ps1:** env 파일을 `/opt/api.env`로 변경 (또는 존재 시 `/opt/api.env`, 없으면 `/home/ec2-user/.env` 사용).

### B. 구조적으로 보강해야 할 항목
1. **Resolver 규칙 3:** X-Tenant-Code fallback을 “허용 Host 또는 ALB/내부 Host”로 제한하는 설정/로직 추가.
2. **is_reservation_cancelled:** 호출부 확인 후, tenant 인자 추가 및 `filter(tenant=tenant, pk=reservation_id)` 적용.
3. **results 집계·기타 `.filter(id=...)` 단독 사용:** 호출 경로가 tenant로 이미 제한된 뒤인지 검토하고, 필요 시 tenant 필터 또는 “선행 조회로 tenant 소속 보장” 문서화.

### C. 문서/규칙/운영 절차로 보강해야 할 항목
1. **docs/02-OPERATIONS/배포.md:** 정석 배포(scripts/v1/deploy.ps1, ECR, /opt/api.env)와 10절 원격 자동 배포 스크립트의 관계 명시. “현재 정석은 deploy.ps1 + ECR이며, deploy_api_on_server.sh는 레거시/수동용” 등.
2. **재발 방지:** 테넌트 경계가 있는 API 추가/수정 시 “tenant 출처 + cross-tenant 조회 가능성” 체크리스트를 09_multitenant_isolation.mdc 또는 CONTRIBUTING에 추가.
3. **QnA 문서:** `docs/02-OPERATIONS/QNA-질문등록-배포에서만-실패-원인.md` 는 현재 resolver 코드와 일치함. 유지.
