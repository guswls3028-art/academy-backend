## 5. 현재 멀티테넌트 구조 요약

### 5.1 tenant resolve

- **규칙 1:** Host가 `TENANT_HEADER_CODE_ALLOWED_HOSTS`(기본 `api.hakwonplus.com`)이고 `X-Tenant-Code` 있음 → `tenant_get_by_code(raw)`.
- **규칙 2:** `request.get_host()` 정규화 → `TenantDomain` 조회 → tenant 반환. 없으면 None.
- **규칙 3:** Host에 TenantDomain 없을 때, **허용 Host 또는 `.elb.amazonaws.com`로 끝나는 Host**일 때만 `X-Tenant-Code`로 fallback (이미 수정 반영됨).
- **bypass:** `/health`, `/healthz`, `/readyz` → tenant=None 허용.
- **그 외:** tenant 없으면 `TenantResolutionError` (404 tenant_invalid 등).

### 5.2 auth binding

- JWT 발급 시 `tenant_code` 필수. `user_get_by_tenant_username(tenant, username)`으로 해당 tenant User만 로그인.
- View 레벨에서 `request.tenant` 사용. 학생앱은 `get_request_student(request)`로 student.tenant 일치 보장.

### 5.3 tenant isolation 구조

- DB는 단일 RDS. 격리는 **애플리케이션 레벨**: 모델에 tenant FK 또는 session→lecture→tenant 경로, selector/view에서 `tenant=` 필터.
- `TenantQuerySet` (`apps/core/db/tenant_queryset.py`) 사용처는 제한적. 대부분 명시적 `filter(tenant=request.tenant, ...)`.

### 5.4 cross-tenant 방지

- Resolver에서 tenant 미식별 시 즉시 실패 (404/403).
- 규칙 3에서 X-Tenant-Code fallback을 허용 Host/ALB로 제한 (이미 반영).

## 6. 현재 배포 구조 요약

### 6.1 배포 스크립트 흐름

- **정석:** `scripts/v1/deploy.ps1` — 빌드 없음(`-SkipBuild`). GitHub Actions OIDC로 ECR 푸시. deploy.ps1은 ECR pull, SSM 갱신, Launch Template UserData 갱신, instance refresh.
- **UserData (api.ps1):** SSM `/academy/api/env` → `/opt/api.env`, `docker pull` + `docker run --env-file /opt/api.env` + 이미지 URI.

### 6.2 원격 검증 흐름

- **run-qna-e2e-verify.ps1:** API ASG 인스턴스 중 **InService 우선** 선택 → SSM Send-Command로 `docker run --env-file /opt/api.env` + `manage.py verify_qna_e2e`. ALB DNS + Host 헤더 사용. PATH 설정 및 InService 우선은 이미 수정 반영됨.
- **run-api-management-remote.ps1:** SSM으로 원격 management command. env 파일 `/opt/api.env` 사용하도록 이미 수정 반영됨.

### 6.3 env / ECR / instance refresh

- SSM `/academy/api/env` (JSON) → UserData에서 `/opt/api.env` 생성.
- ECR full URI (`809466760795.dkr.ecr.ap-northeast-2.amazonaws.com/academy-api:latest`)로 pull/run.
- API ASG instance refresh 시 새 LT 버전으로 새 인스턴스 기동 → 최신 UserData·이미지 반영.

## 7. 확인된 문제 후보

(진단보고서 및 현재 코드 상태 반영. 이미 수정된 항목은 “이미 수정됨”으로 표기.)

### 7.1 영상 진행률 API — Video 조회 tenant 범위

- **문제:** Redis 미스 시 Video를 tenant 없이 조회하거나 다른 테넌트 Redis 사용 가능성.
- **근거:** `apps/support/video/views/progress_views.py` (진단보고서 문제 1).
- **현재 단계:** **이미 수정됨** — Video 조회 시 `session__lecture__tenant=tenant`, cross-tenant fallback 제거된 것으로 이전 세션에서 반영됨. **재검증 권장.**

### 7.2 run-api-management-remote.ps1 env 경로

- **문제:** 원격 management 시 env 파일 경로가 정석 배포(`/opt/api.env`)와 불일치.
- **근거:** `scripts/v1/run-api-management-remote.ps1` (진단보고서 문제 2).
- **현재 단계:** **이미 수정됨** — `/opt/api.env` 통일된 것으로 이전 세션에서 반영됨. **재확인 권장.**

### 7.3 X-Tenant-Code fallback Host 제한 (규칙 3)

- **문제:** Host가 TenantDomain에 없을 때 아무 Host에서나 X-Tenant-Code로 테넌트 결정 가능.
- **근거:** `apps/core/tenant/resolver.py` 규칙 3.
- **현재 단계:** **이미 수정됨** — 허용 Host 또는 `.elb.amazonaws.com`일 때만 fallback 허용된 것으로 이전 세션에서 반영됨. **재확인 권장.**

### 7.4 is_reservation_cancelled tenant 없이 Reservation 조회

- **문제:** `Reservation`을 pk만으로 조회하면 cross-tenant 가능성.
- **근거:** `apps/support/messaging/services.py`, 호출부 `apps/worker/messaging_worker/sqs_main.py`.
- **현재 단계:** **이미 수정됨** — `tenant_id` 인자 추가, 모델에 `tenant_id` 있으면 필터 적용된 것으로 이전 세션에서 반영됨. **재확인 권장.**

### 7.5 results 집계·기타 tenant 선행 제한 미검증
