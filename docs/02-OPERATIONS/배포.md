# 500 배포

**리전**: ap-northeast-2 (서울)  
**원칙**: Video Worker **4GB RAM + 100GB EBS** 필수. 8000 포트는 테스트용만 → 오픈 전 ALB+HTTPS.  
**워커**: ASG Target Tracking — 평소 0대, 큐 쌓이면 증설(Min=0, Max=20). [설계.md](설계.md) 참고.

## 1.1 배포 방식 구분

| 방식 | 설명 | 권장 |
|------|------|------|
| **Formal Deploy (정식 배포)** | `scripts/v1/deploy.ps1` — ECR에 이미 올라간 이미지 기준으로 SSM·Launch Template·API ASG instance refresh. 이미지 빌드·ECR 푸시는 **GitHub Actions만** 수행. main push 시 CI가 ECR 푸시 후 **deploy-api-refresh**로 API ASG instance refresh까지 실행. | 인프라 변경·안정 반영·릴리즈용. |
| **Rapid Deploy (빠른 개발 반영)** | `scripts/v1/api-auto-deploy-remote.ps1` — API 서버에서 **2분마다** main 변경 감지 시 `deploy_api_on_server.sh`로 ECR pull + API 컨테이너만 재시작(ASG instance refresh 없음). 서버에 Git 레포 필요. | 개발 중 빠른 코드 반영용. 작업 종료 시 OFF. |

**상세·비교표:** [DEPLOYMENT-MODES.md](DEPLOYMENT-MODES.md), [FORMAL-DEPLOY.md](FORMAL-DEPLOY.md), [RAPID-DEPLOY.md](RAPID-DEPLOY.md).

## 10. API 서버 Rapid Deploy (원격 제어) — 개발 중 빠른 반영용

API EC2에서 **2분마다** `origin/main` 변경을 감지하면 `scripts/deploy_api_on_server.sh`로 **ECR pull + API 컨테이너만 재시작** (빌드 없음, ASG instance refresh 없음).  
원격에서 ON/OFF는 SSM Send-Command로 가능. **인스턴스에 레포가 없으면** `-RepoUrl` 로 한 번 클론 후 ON 하면 됨.

| 목적 | 명령 (로컬 PowerShell) |
|------|-------------------------|
| **Rapid Deploy ON**  | `pwsh scripts/v1/api-auto-deploy-remote.ps1 -Action On -AwsProfile default` |
| **Rapid Deploy OFF** | `pwsh scripts/v1/api-auto-deploy-remote.ps1 -Action Off -AwsProfile default` |
| **상태·마지막 배포** | `pwsh scripts/v1/api-auto-deploy-remote.ps1 -Action Status -AwsProfile default` |
| **수동 배포 1회**  | `pwsh scripts/v1/api-auto-deploy-remote.ps1 -Action Deploy -AwsProfile default` |

**상세:** [RAPID-DEPLOY.md](RAPID-DEPLOY.md), [Rapid-Deploy-사용법.md](Rapid-Deploy-사용법.md). 작업 종료 시 반드시 OFF.

**서버에서 직접 실행할 때:**

```bash
# 수동 배포 (ECR pull + /opt/api.env + 재시작)
cd /home/ec2-user/academy && bash scripts/deploy_api_on_server.sh

# Rapid Deploy ON (2분마다 main 변경 시 배포)
cd /home/ec2-user/academy && bash scripts/auto_deploy_cron_on.sh

# Rapid Deploy OFF
cd /home/ec2-user/academy && bash scripts/auto_deploy_cron_off.sh
```

## 11. 로컬에서 배포 DB 사용

로컬에서 `python manage.py shell`, `manage.py migrate`, `fix_qna_orphan_created_by` 등을 실행할 때 **배포(프로덕션) RDS/Redis**를 쓰려면, SSM에 저장된 API env를 로컬 `.env`에 반영하면 됩니다.

| 목적 | 명령 (로컬 PowerShell) |
|------|-------------------------|
| **배포 DB/Redis를 .env에 반영** | `pwsh scripts/v1/sync-local-env-from-ssm.ps1 -AwsProfile default` |
| **반영 내용만 확인 (파일 수정 없음)** | `pwsh scripts/v1/sync-local-env-from-ssm.ps1 -AwsProfile default -WhatIf` |

- SSM `/academy/api/env`(JSON)에서 `DB_HOST`, `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_PORT`, `REDIS_HOST`, `REDIS_PORT`, `REDIS_PASSWORD`, `REDIS_DB` 만 읽어와 루트 `.env`에 덮어씁니다. 나머지 키는 그대로 둡니다.
- **주의**: 배포 DB를 쓰면 실제 데이터가 변경됩니다. `migrate`, `fix_qna_orphan_created_by` 등 실행 시 유의하세요.
